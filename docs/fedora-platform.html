Bakgrund

Varför har
Fedora
blockerats?

Fysisk åtkomst till en klient har historiskt inneburit att vi inte kunnat garantera att enheten är oförändrad eller att root-åtgärder är spårbara.

Utan dessa garantier kan vi inte godkänna plattformen för miljöer med krav på integritet och revision.

Lager 1–2

Boot &
fysisk
säkerhet

Första försvarslinjen hindrar en angripare från att ens starta ett alternativt OS eller få tillgång till root-kontot.

Garanterar

En manipulerad eller stulen enhet kan inte startas upp i ett okontrollerat läge. Fysisk åtkomst till hårdvaran räcker inte för att kringgå säkerhetslagren.

BIOS lösenordsskyddat & låst
Bootordning och BIOS-inställningar är låsta med lösenord som bara IT-administrationen känner till. USB- och PXE-boot är inaktiverade.
Säker boot (UEFI Secure Boot)
Endast signerade bootloaders och kernels godkänns av firmware. Modifierat OS kan inte starta.
Okänt lokalt root-lösenord
Det lokala root-kontots lösenord är slumpmässigt genererat och inte känt av användaren. Alla privilegierade åtgärder måste gå via sudo.
Polkit begränsad
Polkit — det grafiska alternativet till sudo — är konfigurerat att bara tillåta specifika underhållsåtgärder, inte godtyckliga root-operationer.
Lager 3 · Dataskydd

Diskkryptering med nätverksberoende upplåsning

LUKS (Linux Unified Key Setup)
Hela disken krypterad. Utan rätt nyckel är innehållet oläsbart — även om disken tas ur enheten.
→
clevis / tang (nätverksupplåsning)
Krypteringsnyckeln hämtas automatiskt från en intern tang-server. Kräver att enheten befinner sig på rätt nätverk.
→
802.1x nätverksautentisering
Nätverksåtkomst kräver ett giltigt klientcertifikat. Utan certifikat — inget nätverk, ingen nyckel, ingen boot.
→
TPM eller smartkort
Certifikatet för 802.1x är bundet till enhetens TPM-chip eller ett personligt smartkort. Kan inte kopieras eller exporteras.
Scenario: disk borttagen

Disken är krypterad med LUKS. Utan tang-serverns nyckel — som bara ges till autentiserade enheter — är innehållet oåtkomligt.

Scenario: enhet stulen

Utan giltig 802.1x-autentisering (TPM/smartkort) får enheten inte nätverksåtkomst och kan därmed inte låsa upp disken.

Scenario: boot från USB

BIOS-lås förhindrar USB-boot. Även om det kringgicks saknas nätverksnyckeln — disken förblir krypterad.

Lager 4 · Internt projekt

sudo-logger
Fullständig
spårbarhet

Varje root-kommando spelas in i realtid och skickas krypterat till en extern loggserver. Inspelningen kan inte stoppas lokalt.

Realtidsöverföring — sessioner skickas direkt, buffras inte lokalt
Fryser vid nätavbrott — sudo blockeras om loggservern är nåbar, ingen loggning = ingen åtkomst
Fullständig sessionsuppspelning — exakt terminalutdata sparas och kan spelas upp i webbgränssnittet
HMAC-signerat — loggposter är kryptografiskt signerade, manipulering detekteras
alice@ws01:~$ sudo dnf update -y
[sudo-logger] session started · alice/ws01_20260313-091422
→ Inspelning aktiv · loggserver bekräftad

alice@ws01:~$ sudo rm -rf /etc/important
[sudo-logger] session started · alice/ws01_20260313-091455
→ Hela sessionen sparad på extern server

~ nätverksavbrott ~
alice@ws01:~$ sudo bash
[ SUDO-LOGGER: log server unreachable — input frozen ]
Waiting for log server to come back...
Säkerhetsgaranti
Root-åtgärder utan aktiv loggning är tekniskt omöjliga. Loggposter på servern kan inte modifieras bakåt i tiden. Revisionskedjan är komplett.
Lager 5–7 · OS-härdning

Systemhärdning
& revision

Oberoende av användarbeteende begränsar OS-nivåkontrollerna vad som är möjligt och dokumenterar allt som sker.

Kombinationseffekt

SELinux begränsar vad processer kan göra. Brandväggen begränsar nätverkstrafik. auditd dokumenterar vad som gjordes — och skickar det omedelbart till SIEM där det inte kan rensas.

SELinux Enforcing mode
Mandatory Access Control på kärnnivå. Processer kan bara komma åt resurser de explicit fått tillstånd till — även root-processer. Begränsar skada vid komprometterade tjänster.
Firewall Lokal brandvägg (firewalld)
Strikt regeluppsättning som bara tillåter godkänd nätverkstrafik. Förhindrar oavsiktliga tjänstexpansioner och laterala rörelser.
auditd Händelseloggning → SIEM
Strikt auditd-konfiguration fångar privilegieeskalering, filsystemsförändringar och systemanrop. Alla händelser vidarebefordras i realtid:
auditd → rsyslog / auditbeat → SIEM (externt) → larm & retention
Lager 8–9 · Perimeterskydd

Externa
hot

Skydd mot att angripare tar sig in via fysiska gränssnitt eller skadlig kod.

Kombinationseffekt

USBGuard stoppar oavsiktlig eller avsiktlig introduktion av skadlig hårdvara. ClamAV hanterar traditionella malware-hot. Tillsammans täcker de de vanligaste fysiska angreppsytorna.

USBGuard — USB-enhetsfiltrering
Bara förgodkända USB-enheter (identifierade via produkt-ID och serienummer) tillåts av systemet. Okända enheter blockeras automatiskt vid inkoppling — inga undantag utan administratörsgodkännande. Skyddar mot BadUSB-attacker och dataexfiltrering via USB.
ClamAV — antivirusskydd
Kontinuerlig skanning av filsystemet och inkommande filer. Signaturuppdateringar sköts automatiskt. Komplement till SELinux-restriktionerna för känd skadlig kod.
Polkit — begränsad systembehörighet
Polkit styr vilka privilegierade åtgärder som grafiska applikationer kan utföra utan lösenord. Konfigurerat till minimala rättigheter: bara specifika underhållsoperationer tillåts. Administrativa root-åtgärder kräver alltid sudo — och loggas av sudo-logger.
