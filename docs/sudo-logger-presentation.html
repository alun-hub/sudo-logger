The Problem

Privileged access is your biggest blind spot

🕵️

No Accountability

Standard sudo logs only what command was run — not what the operator actually did inside a root shell. A 4-hour sudo bash is invisible.

💨

Logging Can Be Skipped

Traditional logging relies on the client cooperating. An attacker with root can simply kill the logging daemon and continue working undetected.

🏝️

Siloed, Unanalysed

Even when logs exist, they live on individual hosts in syslog format. No central replay, no risk scoring, no anomaly detection, no SIEM integration.

🔓

No Enforcement

Nothing prevents a privileged user from editing /etc/sudoers, killing the audit daemon, or replacing the logging agent with a dummy binary.

🖥️

GUI Apps Unrecorded

sudo gedit /etc/passwd produces no terminal output. Graphical sudo sessions are completely invisible to text-based auditing tools.

🔑

Secrets in Plain Sight

API keys, AWS credentials, and passwords typed during sudo sessions end up verbatim in log files accessible to log administrators.

Compliance reality: SOC 2, ISO 27001, PCI-DSS, and HIPAA all require privileged access to be monitored and auditable. A syslog entry showing sudo bash does not satisfy this requirement.

Feature 01

Fail-Closed Session Recording

How it works

User runs sudo → plugin connects to agent (Unix socket) → agent opens TLS connection to log server If server unreachable: SESSION_ERROR → sudo is blocked. Command never runs. If server accepts: SESSION_READY → sudo forks child process → ALL I/O streamed in real time → server signs every chunk (ed25519) → agent verifies signature before caching ACK

The ACK loop

The plugin polls the agent every 150 ms for the latest ACK timestamp. The agent answers from its in-memory cache — sub-millisecond, no server round-trip needed.

If ACKs go stale for ~800 ms → session cgroup is frozen. The user's terminal is suspended at the kernel level. No job-control signal can escape this.

What is captured

  • Terminal input — every keypress the operator typed
  • Terminal output — everything they saw on screen
  • stdin / stdout / stderr for non-tty sessions
  • Exact timing — nanosecond precision for faithful replay
  • Session metadata — user, host, command, CWD, terminal size
  • Exit code — written by server on clean session end

Security guarantees

  • Sudo blocked at start if server is unreachable
  • Session frozen within 800 ms of network loss
  • Ctrl+C always works — user is never permanently stuck
  • ed25519-signed ACKs — network attacker cannot forge freeze release
  • ACK bound to session ID — replay attacks impossible
  • Host verified against TLS certificate CN/SAN

Format

Sessions stored as asciinema v2 on the log server — tamper-evident, replay-compatible, never accessible to the users being recorded.

Feature 01 — Deep Dive

The Freeze Mechanism — No Escape

cgroup.freeze

The agent places every sudo session in a dedicated cgroup subtree. On network loss: echo 1 > cgroup.freeze suspends all processes atomically at the kernel level.

Cannot be escaped with fg

CLONE_NEWCGROUP

The plugin calls unshare(CLONE_NEWCGROUP) after session start. Children see the session cgroup as filesystem root — /sys/fs/cgroup/../../escape resolves inside the subtree only.

Even CAP_SYS_ADMIN cannot escape

SIGSTOP Fallback

GUI apps that GNOME/systemd moves to foreign cgroups before the namespace is set up are tracked via /proc/<pid>/cgroup polling and frozen with SIGSTOP/SIGCONT.

Process-group safe

⏱ Freeze Timeout

If the network never recovers (TCP timeout expired), the agent's watchdog fires after 3 min (configurable). It unfreezes the cgroup, sends FREEZE_TIMEOUT to the plugin, and terminates the session gracefully.

The UI shows an ⏱ amber badge — a network outage is not a security incident.

🕵️ Incomplete Detection

If the agent is killed mid-session, the server writes an INCOMPLETE marker and logs a SECURITY: warning. The plugin detects the socket drop (EPIPE) and sends SIGTERM within 150 ms — the session is terminated, not silently lost.

The UI shows a ⚠ red badge and adds +15 to risk score.

Feature 02

eBPF Session Recording

The agent attaches three eBPF tracepoints to the running kernel at startup. No kernel patches, no recompilation — CO-RE (Compile Once, Run Everywhere) via BTF.

sys_enter_write

Captures TTY I/O from all processes in tracked cgroups. SSH login sessions, pkexec scopes, and interactive sudo sessions are all captured. su, screen, and tmux output is captured transparently within their enclosing login session.

sys_enter_execve

Detects sudo and pkexec invocations for divergence tracking — verifies that every executed sudo also has a corresponding plugin session.

sched_process_exit

Closes eBPF sessions when the tracked process exits, freeing kernel BPF map entries.

What eBPF captures that the plugin cannot

  • SSH login session I/O (entire terminal session, not just sudo)
  • pkexec sessions (PolicyKit privilege escalation)
  • su invocations inside sudo shells
  • tmux / screen multiplexer sessions within a sudo context
  • Non-interactive background pkexec commands

Divergence Detection Security

If eBPF sees a sudo execve but no matching plugin SESSION_START arrives within 30 seconds, the agent sends a DIVERGENCE_ALERT to the server.

The replay UI shows this as a ⚠ no plugin session — indicating sudo ran without logging. This catches:

  • A replaced or disabled plugin .so
  • A missing or corrupt Plugin line in /etc/sudo.conf
  • Sudo invocations before the agent started
Graceful degradation: if BTF support is absent (older kernels), the agent starts in plugin-only mode. All other functions continue normally.

Feature 03 — Process Sandbox

eBPF LSM: Root Cannot Escape

The process sandbox attaches 13 eBPF hooks to the Linux Security Module framework. Every protected operation is intercepted in the kernel before it reaches the filesystem or process — no userspace code can intercept or bypass this.

The fundamental problem it solves: a user with sudo bash is effectively root. They can write evil ALL=(ALL) NOPASSWD:ALL directly to /etc/sudoers — modifying the very rules that restrict them.
With the sandbox: that write returns EPERM. The kernel rejects it. The session continues — only the write is denied. The violation is logged and sent to the SIEM.

Dual scoping — catches cgroup races

Two complementary mechanisms ensure even short-lived commands like sudo pkill auditd are caught:

  • sandboxed_cgroups — cgroup ID registered at session start
  • sandboxed_pids — sudo PID tracked; fork hook propagates to all descendants atomically, before the child runs any userspace code

Btrfs-correct device IDs

Device IDs are read from /proc/self/mountinfo field 3 — matching exactly what the kernel BPF program reads from inode→i_sb→s_dev. Works correctly on Btrfs subvolumes where stat().st_dev and i_sb→s_dev differ.

Feature 03 — Sandbox Configuration

What Can Be Protected

📄 Files

Prevent write, delete, truncate, rename, and attribute changes on sensitive files and directories.

files: - /etc/sudoers - /etc/sudoers.d # protects directory contents - /etc/sudo-logger # the agent's own config - /etc/pam.d - /etc/ssh/sshd_config

💿 Devices

Block raw disk and memory access that could be used to bypass filesystem-level controls.

devices: - /dev/mem - /dev/kmem - /dev/sda - /dev/nvme0n1

⚙️ /proc Entries

Prevent modification of sensitive kernel parameters via procfs.

proc: - /proc/sysrq-trigger - /proc/sys/kernel/ngroups_max - /proc/sys/kernel/randomize_va_space

🔌 Sockets

Prevent deletion and replacement of Unix domain socket files — protecting agent communication channels from interception.

sockets: - /run/sudo-logger/plugin.sock - /run/dbus/system_bus_socket

🛑 Processes

Prevent sending ANY signal (including SIGKILL) to protected daemons. Effectively prevents systemctl stop auditd or pkill sshd from a sudo session.

processes: - sudo-logger-age # the agent itself - auditd # Linux audit daemon - sssd # auth/directory services - sshd # OpenSSH - firewalld # firewall - systemd-journal # journald (15 char limit)

Feature 05

Centralized User Blocking

The problem: an incident is detected. You need to immediately revoke sudo access for a compromised account — across 200 hosts — without touching sudoers files on every machine.

How it works

Security team → Blocked Users tab in browser → PUT /api/blocked-users → policy stored (yaml / PostgreSQL) Log server (every 30 s, or immediate) → reloads blocked-users policy Next sudo attempt by blocked user → SESSION_DENIED during startup handshake → agent forwards SESSION_DENIED to plugin → plugin shows red banner to user → sudo exits. Command never runs.

Capabilities

  • Block globally (all hosts) or per specific hosts
  • Configurable block message shown to user
  • Internal reason/ticket reference for audit
  • Policy reloaded every 30 s — no restart needed
  • Managed via browser UI or YAML file
  • In Kubernetes: stored in PostgreSQL, shared across all replicas automatically
Enforcement is server-side. A compromised or modified agent on the client cannot bypass a block — the denial happens during the log server's startup handshake, before sudo forks any child.

Feature 09

Wayland Screen Capture

GUI programs started with sudo produce no terminal I/O — they would be completely invisible to text-based recording. sudo-logger solves this without any compositor patches.

A transparent Wayland proxy is spawned between the sudo'd application and the compositor. The proxy intercepts wl_surface_commit calls and JPEG-encodes each frame. The replay interface shows them as an image slideshow.
  • No compositor patches required
  • Works with KDE, GNOME, Sway, and any Wayland compositor
  • Multiple GUI apps can be recorded in sequence in one session
  • Capture interval configurable (default 300 ms)
  • Disable per-host in agent.conf when not needed
Legacy X11: remove DISPLAY from sudo environments via Defaults env_delete += DISPLAY to make unrecorded GUI sessions fail-closed.

Features 11 + 12

SELinux Confinement & Idle Timeout

🔐 SELinux Domain Confinement

The agent runs as sudo_shipper_t under Fedora/RHEL's targeted SELinux policy. The policy module ships in the client RPM and is loaded automatically on install (semodule -i in %post).

  • Kernel-level restrictions on agent process capabilities
  • Cannot read files outside its expected paths
  • Cannot bind to arbitrary network ports
  • Runs enforcing without any boot parameter changes on Fedora 38+
  • restorecon applied in %posttrans for correct labels
Defense-in-depth: even if the agent binary were replaced, SELinux would prevent the replacement from accessing files or network resources the real agent is not supposed to touch.

⏰ Idle Session Timeout

Sessions where the user has walked away from the terminal stay open indefinitely, holding a freeze-capable connection and consuming log server resources. The idle timeout closes them automatically.

# /etc/sudo-logger/agent.conf idle_timeout = 30m
  • Triggered by absence of user input (stdin/tty-in) for the configured duration
  • Sends SIGHUP to the sudo process — terminates the session gracefully
  • Session is closed and written to the log server
  • Configurable per-host; disabled by default
  • Uses Go duration syntax: 30m, 2h, etc.
Idle timeout is distinct from freeze timeout — it applies to healthy, connected sessions, not frozen ones.

Security Architecture
A layered defense model. Every trust boundary is explicit. Every control is verifiable.
Security Properties

Defense in Depth

🚧
Fail-Closed
If the log server is unreachable when sudo runs, the session is rejected before the command executes. No sudo activity can occur without a live server connection.
🧊
Child Process Frozen on Network Loss
Within ~800 ms of the last ACK/heartbeat, the session cgroup is frozen. No command input reaches the running process until the server responds.
🔒
Freeze Cannot Be Escaped
unshare(CLONE_NEWCGROUP) in the plugin at session start. All children see the session cgroup as filesystem root. Even CAP_SYS_ADMIN cannot write to a parent cgroup from inside the namespace.
🔑
Asymmetric ACK Signing (ed25519)
Server signs each ACK with its private key over sessionID ‖ seq ‖ ts_ns. Private key never leaves the server. A compromised client cannot forge ACKs. A captured ACK cannot be replayed to unfreeze a different session.
🪪
Host Identity Verified Against TLS Certificate
Server rejects SESSION_START if claimed host does not match the CN or DNS SANs of the client certificate. A compromised agent on host A cannot forge log entries attributed to host B.
🔌
Plugin Socket Peer Verification
Agent verifies via SO_PEERCRED that only root processes (sudo) may connect to the Unix socket — beyond file permission checks.
🎲
Session ID Collision Resistance
Session IDs include nanosecond precision and 4 cryptographically random bytes. Simultaneous sessions on the same host are always distinct.
🗂️
Tamper-Evident Log Storage
Logs are written on a separate server the sudo-running users have no access to. Log directory paths are validated with EvalSymlinks to prevent path traversal.
🛡️
eBPF LSM Process Sandbox
13 kernel hooks enforce a deny-list. Not bypassable by root, CAP_SYS_ADMIN, or any userspace technique. Violations generate alerts sent to the session's SIEM endpoint.
⚡
Active Session Terminated If Agent Dies
Plugin detects socket drop (EPIPE/ECONNRESET/EOF) within 150 ms and sends SIGTERM to sudo. The attacker cannot continue working unlogged. New sudo sessions are fail-closed until the agent restarts (~2 s).
🔍
Input Validated Before Filesystem Use
User, host, and session ID fields validated with strict regexes. Cgroup names validated before directory creation. Path confinement verified with symlink resolution.
Threat Model

What Can an Attacker Do?

Attack vectorMitigationResidual risk
Kill the agent binary Plugin detects socket drop → SIGTERM within 150 ms. Agent restarts in 2 s. Remaining sessions fail-closed. The kill command itself is already in the log
Replace agent binary Sandbox protects agent binary path. SELinux restricts replacement's capabilities. Replacement survives reboot without sandbox; physical access needed
Edit /etc/sudoers Sandbox LSM blocks the write with EPERM even for root. Violation alert sent to SIEM. None within an active sudo session
Kill auditd / sshd Sandbox lsm/task_kill blocks SIGKILL. systemctl stop fails via SIGTERM block. None within an active sudo session
Forge ACKs to unfreeze ed25519 — private key on log server. Client only holds public key. None — asymmetric crypto
Drop network to unfreeze Network drop triggers freeze (not unfreeze). Freeze timeout terminates session. ~800 ms of unrecorded activity before freeze
sudo with no plugin eBPF divergence detection — alert sent within 30 s if no SESSION_START follows execve. Session content not captured, but activity flagged
Escape session cgroup CLONE_NEWCGROUP + trackDescendants + aux cgroup registration. PID tracked at fork. Escape to foreign cgroup before namespace set up (PAM race) — covered by sandboxed_pids
