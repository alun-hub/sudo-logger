The hidden problem with standard sudo

Root can erase its own tracks

Most sudo tooling logs to the same host the user runs sudo on. An administrator with sufficient privileges can delete, stop, or bypass logging entirely — making the audit trail an illusion of control.

🗑️

Logs live on the same disk

Standard sudo writes to /var/log/auth.log or auditd on the same host. Root can delete them, stop syslog, or overwrite them — before your SIEM has even collected them.

rm -rf /var/log/auth* && systemctl stop rsyslog
📡

Network drop = audit gap

If the connection drops mid-session the process keeps running without logging. You never know what happened during the outage. There is no record that a gap even occurred.

Session ran 14 min unlogged after network fault
👻

Fail-open by design

Standard tools let sudo run even when the log server is down. Logging is treated as "nice to have" rather than a hard requirement — opening a window that attackers can exploit.

Log server down 3 min = 47 unlogged sudo commands
Four guarantees no other tool provides

Designed to be impossible to bypass

Every security property is enforced in the Linux kernel or via the mTLS protocol — not by userspace policies that root can disable.

01

Fail-closed — always

If the log server is unreachable when sudo starts, the session is rejected before the command executes. No exceptions. An attacker cannot simply wait out the log server to get an unlogged window.

No log server = no session. Period.
02

Kernel freeze on network loss

If the ACK heartbeat is absent for 800 ms, the agent freezes all processes in the session via cgroup.freeze — without job-control signals, so fg cannot break out. CLONE_NEWCGROUP isolation prevents CAP_SYS_ADMIN from migrating to another cgroup to escape.

Freeze within ~800 ms · auto-recovery · 3 min timeout
03

20 kernel hooks — root cannot hide

The eBPF LSM Sandbox attaches 20 hooks (16 LSM + 2 tracepoints). They block writes to protected files, the bpf() syscall, AF_NETLINK (firewall tampering), ptrace(), mount() and CAP_AUDIT_CONTROL — all scoped per session cgroup. Root cannot swap the agent, kill auditd, or delete logs.

Kernel-enforced · not bypassable even by root
04

eBPF ring buffer — every keystroke

Three eBPF tracepoints capture TTY I/O from all processes in tracked cgroups — not just direct sudo children. su, screen, tmux and pkexec are captured transparently. If sudo runs without the plugin, a ⚠ no plugin alert appears in the replay UI.

Ring buffer · sys_enter_write hook · divergence detection
What's included

Complete privileged access platform

From kernel-level hardening to centralized sudoers management and SIEM integration — everything in one open-source package under AGPLv3.

🔌
Sudo I/O Plugin

Loaded directly by sudo as a native I/O plugin via /etc/sudo.conf — no wrappers or LD_PRELOAD. Captures stdin, stdout, stderr, tty-input, tty-output, environment variables, and the executed command. Everything is streamed in real time with no local buffering on the client. The plugin detects a socket drop within 150 ms and sends SIGTERM to the session.

C pluginsudo APIreal-time stream
🔒
mTLS + Ed25519 ACK Signing

Client and server authenticate each other with certificates. Every received chunk is confirmed with an Ed25519-signed ACK — the agent refuses to persist data that has not been ACK'd by the server. A heartbeat is sent every 400 ms; the session is frozen if an ACK is missing for 800 ms. Replay attacks and silent data loss are structurally impossible.

mTLSEd25519400 ms heartbeat
🛡️
eBPF LSM Process Sandbox

20 kernel hooks (16 LSM + 2 tracepoints) enforce a deny-list inside the Linux kernel. Blocked operations include writes to protected files and devices, kill signals to protected processes, the bpf() syscall, AF_NETLINK, ptrace, mount, and dangerous capabilities. All enforcement is scoped per session via cgroup-ID with PID tracking propagated atomically at every fork.

eBPF LSMkernel-enforcedcgroup scoped
🕵️
eBPF Divergence Detection

Three eBPF tracepoints (TTY I/O, execve, process-exit) compare kernel-observed activity against what the plugin reported. If sudo runs without the plugin — e.g. after an unauthorized modification to /etc/sudo.conf — a ⚠ no plugin alert is shown in the replay UI. Also captures pkexec invocations inside sessions and flags them as separate events.

tracepointsdivergence alertpkexec detection

Session Replay UI

A React + TypeScript web interface for searching, filtering, and replaying recorded sudo sessions as asciinema recordings with full terminal fidelity. Sort by date, user, host, duration, or risk score. Filter by date range. The bottom info bar shows start time, duration, working directory, and a copyable session ID. Download raw .cast files for offline analysis.

Reactasciinemafull-text search
⚠️
Automatic Risk Scoring

Every session is scored against configurable YAML rules at ingest time. Rules match on command patterns (regex), user, host, and flags such as after_hours or incomplete (session terminated without clean exit). Risk levels: Critical, High, Medium, Low, Info. The replay UI color-codes sessions and surfaces matched rules. Rules hot-reload without restart.

YAML ruleshot-reloadregex
🔍
Automatic Secret Redaction

The agent masks AWS access keys, API tokens, Bearer headers, JWT tokens, URL passwords, and other secrets in the terminal stream before they reach the log server. Custom regex patterns can be added via mask_pattern in agent.conf. Redaction happens in memory on the monitored host — secrets never leave the machine in plaintext.

in-memoryconfigurablepre-transmission

SIEM Integration

Forward risk alerts in real time to any SIEM in JSON, CEF (ArcSight/QRadar), or OCSF format via HTTPS or syslog. Configuration hot-reloads every 30 seconds without restart. A Prometheus metrics endpoint is included for Grafana dashboards covering session counts, risk distribution, streaming latency, and agent connectivity.

Just-In-Time Approvals

Require justification and real-time approval before a sudo session starts. The session is frozen in the kernel via cgroup.freeze while the request waits. Supports Mattermost and Slack with interactive Approve/Deny buttons, time-limited approval windows, and automatic TTL closure. Approval state is persisted so a server restart does not abandon waiting sessions.

OPA Policy Engine

Replace hard-coded allow/deny lists with a full policy language based on Open Policy Agent and Rego. Rules can match on user, host, command, OS group (local/SSSD/LDAP/AD), day of week, and time of day. Priority: deny > allow > challenge. Changes hot-reload without restart. The replay UI includes a policy editor with live syntax validation.

OPA / RegoSSSD / ADhot-reload

Centralized Sudoers Management

Push sudoers rules to all managed hosts from the replay UI. The agent validates and applies changes atomically via visudo -c and reports sync status in real time. Supports a global default template with per-host overrides, a visual rule editor, and a raw editor with syntax highlighting. Every deployed sudoers file is snapshotted for audit purposes.

Scale log servers horizontally in Kubernetes. Sessions are stored in S3-compatible object storage, metadata in PostgreSQL, with built-in leader election for SIEM forwarding. Designed for 500+ simultaneous sessions with asynchronous batch disk I/O and a zero-deadlock architecture. Local flat-file storage is the default for single-node deployments.

How it works

Simple deployment, strong security

A lightweight plugin on each host sends encrypted session data to a central log server over mTLS. Nothing needs to be installed on clients — sudo integration is transparent via /etc/sudo.conf.

Getting started

Up and running in minutes

Install the plugin on monitored hosts, deploy the log server, and you have tamper-proof sudo auditing. See the full documentation for TLS setup, Helm charts, and configuration reference.

# --- On each monitored host ---
sudo dnf install sudo-logger-client

# /etc/sudo.conf — load the plugin
Plugin sudoreplay /usr/libexec/sudo/plugin.so \
  logserver=logserver.corp.local:9876 \
  cert=/etc/sudo-logger/client.crt \
  key=/etc/sudo-logger/client.key \
  ca=/etc/sudo-logger/ca.crt

# --- On the log server ---
sudo dnf install sudo-logger-server sudo-logger-replay
sudo systemctl enable --now sudo-logger-server sudo-logger-replay

# Deploy with Helm (Kubernetes)
helm install sudo-logger ./helm/sudo-logger \
  --set tls.existingSecret=sudo-logger-tls
