Executive Summary

sudo-logger is the industry's most advanced audit recording solution for Linux privileged access. While traditional auditing systems capture only the execution event, sudo-logger records the entire interactive session as a high-definition, cryptographically signed stream.

Designed for the Zero-Trust Enterprise, the system ensures absolute Chain of Custody by evacuating audit data from monitored hosts in sub-second real-time. This prevents even a root user from altering or concealing their actions.

Enterprise Grade: sudo-logger is built to scale to tens of thousands of nodes, supporting S3-compatible object storage, PostgreSQL metadata backends, and full OpenID Connect (OIDC) single sign-on.

System Architecture

The suite follows a micro-services architecture to isolate security boundaries and maximize ingestion availability.

1. Sudo Plugin (C)

A native sudo_plugin implemented in C. It hooks into the terminal I/O loop of the sudo binary. By operating at the TTY level, it captures data before it can be obfuscated by sub-shells or interactive editors.

2. Agent (Go)

The host-level security orchestrator (sudo-logger-agent). It is responsible for:

  • Local Identity Proof: Correlating TTY activity with D-Bus/Polkit session tokens.
  • eBPF Enforcement: Attaching syscall hooks to detect and block pkexec or su bypasses.
  • Secure Streaming: Encrypting and signing data for transmission over mutual TLS (mTLS).
  • cgroup v2 Control: Freezing session execution if the central audit server is unreachable.

3. Log Server (Go)

The central ingestor (sudo-logserver). A high-performance, stateless service that validates Agent identities and persists signed audit chunks to S3 or local storage.

4. Replay Server (Go)

The administrative control plane. It hosts the Web UI, manages the JIT approval lifecycle, and provides an API-first interface for SIEM integration.

Fail-Secure Mechanism: The system maintains a tight ACK loop between the Plugin and Ingestor. If ACKs are missing for more than 2 seconds, the Plugin locks the terminal using cgroup freezing, preventing unlogged activity during network outages.

Session Lifecycle

The technical sequence of events for an audited privileged command.

  1. Invocation: User runs sudo dnf update. Sudo loads the plugin.
  2. Plugin Handshake: The plugin connects to the Agent via a protected Unix socket.
  3. Context Discovery: The Agent identifies the user via D-Bus and records the parent PID, environment, and TTY path.
  4. Remote Auth: The Log Server is queried. If the command matches a JIT rule, the user is challenged for a "Reason".
  5. Approval: The command is held in a SIGSTOP state until an auditor clicks "Approve" in the Web UI.
  6. Recording: Every byte of TTY input/output is bundled into signed binary chunks.
  7. Risk Scoring: As data arrives, the Server analyzes it against risk-rules.yaml. If a forbidden string (e.g. nc -l) appears, the risk score is instantly elevated.
  8. Finalization: Sudo exits. Final chunks are flushed. The session is closed and indexed for search.

Log Integrity & Ed25519

In a high-security environment, logs must be immutable. sudo-logger uses Ed25519 signatures to provide non-repudiation.

  • Continuous Verification: Every single chunk of data is signed. A missing or reordered chunk will immediately break the cryptographic chain.
  • Hardware Correlation: Signatures are generated using a private key unique to the monitored host, proving the data originated from that specific node.
  • Audit Tooling: The verify-integrity.sh script allows forensic teams to validate log files offline, independent of the Replay Server.

Deployment: RPM Installation Guide

Optimized for RHEL 8/9, AlmaLinux, Rocky Linux, and CentOS Stream.

1. Repository Configuration

# /etc/yum.repos.d/sudo-logger.repo
[sudo-logger]
name=sudo-logger Enterprise
baseurl=https://packages.company.com/rpm/rhel/\$releasever/\$basearch/
enabled=1
gpgcheck=1
gpgkey=https://packages.company.com/RPM-GPG-KEY-sudologger

2. Client (Monitored Host)

# Install agent and plugin
dnf install sudo-logger-client

# Configure server endpoint in /etc/sudo-logger/agent.conf
# server = ingestor.audit.internal:9876

# Enable services
systemctl enable --now sudo-logger-agent

3. Server (Control Plane)

# Install ingestor and UI
dnf install sudo-logger-server sudo-logger-replay

# Configure /etc/sysconfig/sudo-logserver
# systemctl enable --now sudo-logserver
# systemctl enable --now sudo-replay-server

Deployment: Kubernetes Orchestration

Kubernetes is the recommended architecture for Distributed Storage Mode, enabling horizontal scaling to thousands of agents.

Architecture Components

  • Log Ingestors (Deployment): Stateless pods running sudo-logserver.
  • Replay Portal (Deployment): Pods running sudo-replay-server.
  • Storage: PostgreSQL for metadata; S3-Compatible bucket for binary logs.

Configuration Strategy

Use Kustomize to inject production secrets and storage URLs.

# k8s/overlays/prod/kustomization.yaml
resources:
  - ../../base
configMapGenerator:
  - name: server-config
    literals:
      - DB_URL=postgres://user:pass@pg-prod:5432/audit
      - S3_BUCKET=audit-recordings

Legacy Data Migration

Move your existing recordings from local disk (v1) to a Distributed Cloud (v2) architecture. 

sudo-logger-migrate \
  -logdir /var/log/sudoreplay \
  -storage distributed \
  -db-url "postgres://..." \
  -s3-bucket "audit-v2" \
  -s3-endpoint "https://minio.internal:9000"

The tool iterates over all .cast files, re-calculates risk scores, uploads the binary blobs to S3, and populates the PostgreSQL database.

Log Server (Ingestor) Flags

CLI flags usually managed in /etc/sysconfig/sudo-logserver.

Core Configuration

  • -listen: TLS address for Agent streams (default :9876).
  • -health-listen: HTTP port for /healthz and /metrics.
  • -signkey: Path to Ed25519 PEM private key for ACK signing.
  • -ca: CA used to verify Agent certificates.
  • -strict-cert-host: If true, Agent hostname must match Cert CN.

Storage Backend

  • -storage: local (flat files) or distributed (S3+Postgres).
  • -db-url: PostgreSQL DSN.
  • -s3-bucket / -s3-endpoint / -s3-region: S3 settings.
  • -s3-path-style: Use path-style URLs (required for MinIO).
  • -buffer-dir: Disk buffer for S3 uploads.

Replay Server (Web UI) Flags

CLI flags managed in /etc/sysconfig/sudo-replay-server.

UI & Security

  • -listen: Address for the web portal (default :8080).
  • -tls-cert / -tls-key: Enable HTTPS for the UI.
  • -admin-users: Comma-separated list of local/OIDC administrators.
  • -trusted-user-header: Trust identity from upstream proxy (e.g. X-Forwarded-User).
  • -rules: Path to risk-rules.yaml.
  • -siem-config: Path to siem.yaml.

SSO Integration

  • -oidc-issuer: OIDC provider URL (e.g. https://accounts.google.com).
  • -oidc-client-id / -oidc-client-secret: Provider credentials.
  • -oidc-redirect-url: Public callback URL.

Web UI: JIT Approval Workflows

Enforce the "Four-Eyes" rule for high-risk infrastructure changes.

Approval Lifecycle

  1. User Challenge: User runs sudo rm -rf /data. Terminal is held.
  2. Justification: User enters a reason (e.g. "Ticket #123").
  3. Notification: Alerts sent to Slack/Mattermost.

  5. Decision: One-click Approve or Deny. Denials can include feedback to the user.

YAML Reference: Risk Rules Engine

Located at /etc/sudo-logger/risk-rules.yaml. Defines the scoring logic.

Rule Fields

FieldTypeDescription
idstringUnique rule identifier.
scoreintRisk points to add (0-100).
reasonstringDescription shown in UI.
commandMatchPatternMatches the sudo command line.
contentMatchPatternMatches interactive terminal output.
command_base_any[]stringBasename matches (e.g. nc).
runasstringMatch target user identity.
after_hoursboolMatch if started 23:00-06:00.
incompleteboolMatch crash events.
min_durationfloatMin seconds to match.

MatchPattern Syntax

# Rule within risk-rules.yaml
- id: secret_access
  score: 95
  reason: Attempt to read system secrets
  content:
    contains_any: ["/etc/shadow", "/etc/gshadow"]
    also_any: ["cat", "grep", "vi"] # AND cat/grep/vi was used

YAML Reference: eBPF Sandbox Policy

Located at /etc/sudo-logger/sandbox.yaml. Enforces host-level blocking.

Policy Deny-List

deny:
  - pattern: "/usr/bin/nc*"
    message: "Network exfiltration tools are forbidden."
  - pattern: "/usr/bin/python*"
    message: "Scripting interpreters are blocked during audit."
  - pattern: "/usr/bin/gdb"
    message: "Debuggers are prohibited in this session."
Kernel Enforcement: Unlike shell aliases, eBPF sandbox policy is enforced by the kernel. Renaming or symlinking a binary will NOT bypass the policy.

Security: eBPF Syscall Interception

Bypassing a sudo plugin is trivial for a root user (e.g., using pkexec). sudo-logger prevents this using eBPF Tracepoints.

The Agent attaches to the sys_enter_execve kernel hook. Every process started on the system is analyzed. If the process is a child of an active sudo-logger session, its execution is intercepted, logged, and checked against the sandbox policy.

Security: D-Bus & Polkit Auditing

Privilege escalation via D-Bus (pkexec, systemctl) is fully audited. The Agent monitors the system bus and captures the caller's identity, the requested action, and the authorization decision.

Security: Immutable Auditd Rules

The audit/99-immutable.rules file locks the Linux audit configuration. Once loaded, even a root user cannot disable auditing or delete rules until the next reboot.

Security: System Hardening

  • RefuseManualStop: The Agent service cannot be stopped via systemd by default.
  • Capabilities: Components drop all unnecessary privileges after startup.
  • Memory Safety: All high-level logic is implemented in memory-safe Go.
