## Build system

### Prerequisites

```bash
sudo dnf install clang llvm libbpf-devel bpftool golang
```

The C plugin's `sudo_plugin.h` header is vendored in `plugin/include/` (verify
with `ls plugin/include/`), so no `sudo-devel` package is required to build
the plugin from source — `plugin/Makefile` compiles with `-Iinclude`, pointing
at the vendored copy, not a system header. (The `sudo-logger-client` RPM spec
still lists `BuildRequires: sudo-devel`, but the actual `%build` step does not
reference anything from that package — it only uses the vendored header.)

For documentation:

```bash
sudo dnf install pandoc
pip install mkdocs mkdocs-material
```

### First-time setup (eBPF vmlinux.h)

The eBPF programs require a `vmlinux.h` file generated from the running
kernel's BTF (BPF Type Format) data. This file is not committed to the
repository:

```bash
make vmlinux-agent      # generates go/cmd/agent/bpf/vmlinux.h from /sys/kernel/btf/vmlinux
make generate-agent     # runs bpf2go to compile recorder.c and sandbox.bpf.c → Go bindings
```

`vmlinux-agent` calls `bpftool btf dump file /sys/kernel/btf/vmlinux format c`
and writes the output to `go/cmd/agent/bpf/vmlinux.h`. Re-run this step after
upgrading the kernel.

`generate-agent` runs `go generate ./cmd/agent/` which invokes `bpf2go` to
cross-compile the eBPF C sources for the current architecture (x86_64 or
arm64, derived from `uname -m`) and generates the `recorder_bpfel.go` and
`sandbox_bpfel.go` Go binding files.

### Building the agent

```bash
make agent
# equivalent: cd go && go build -o ../bin/sudo-logger-agent ./cmd/agent/
```

The `agent` target depends on `generate-agent`, which depends on
`vmlinux-agent`. On a fresh checkout, `make agent` runs the full chain.

The resulting binary is written to `bin/sudo-logger-agent`.

### Building the log server and replay server

`go/Makefile` is a separate, smaller Makefile from the top-level one used for
the agent (that one lives at the repo root and has the `agent`/`vmlinux-agent`
targets described above). Run these from the `go/` directory:

```bash
make server         # go build -o cmd/server/server ./cmd/server/
make replay-server   # runs ui-build (npm ci && npm run build) first, then
                     # go build -o cmd/replay-server/replay-server ./cmd/replay-server/
make all             # builds both replay-server and server
```

There is no `make replay` target — it is named `replay-server`. Binaries land
inside their own `cmd/` subdirectory (`go/cmd/server/server`,
`go/cmd/replay-server/replay-server`), not in a shared `bin/` directory like
the agent. The RPM specs do not use `go/Makefile` at all — they invoke
`go build -mod=vendor -o sudo-logserver ./cmd/server` and
`go build -mod=vendor -o sudo-replay-server ./cmd/replay-server` directly,
which is where the `sudo-logserver`/`sudo-replay-server` binary names actually
come from.

Because `replay-server` depends on `ui-build`, `make replay-server` (or
`make all`) builds the frontend automatically — you do not need to build it
separately first. The replay server binary embeds `cmd/replay-server/static/`
using `//go:embed`.

### Building the plugin (C)

```bash
make -C plugin        # gcc → sudo_logger_plugin.so
make install -C plugin # installs to /usr/libexec/sudo/
```

The plugin Makefile uses `gcc` (not clang) and links against the vendored
`plugin/include/sudo_plugin.h` header via `-Iinclude`. The output is
`plugin/sudo_logger_plugin.so`.

### Building the frontend (React SPA)

```bash
cd go/cmd/replay-server/ui
npm install
npm run build   # tsc -b && vite build → outputs to ../static/
```

Vite is configured (in `vite.config.ts`) to write the production build to
`../static/`, which is the directory embedded by the Go server. The dev
server proxies `/api` requests to `:8080` so it can be used against a running
replay-server instance.

### Building documentation

```bash
make docs-manual    # pandoc → docs/manual.html (also copied to replay-server static/)
make docs-site      # mkdocs build → docs/site/
```

`docs-manual` concatenates the chapter Markdown files and runs pandoc to
produce a single-file HTML manual. The output is also copied into the replay
server's static directory so auditors can access it from the UI.

## Building RPMs

### Version scheme

Three packages are versioned independently. Each has its own `Version:` and
`Release:` fields in its spec file.

- Increment `Version:` for code changes that affect that package's binary.
  Reset `Release:` back to `1%{?dist}` each time `Version:` changes.
- Increment `Release:` only for spec-only fixes (permissions, scriptlets,
  dependencies) where the source code is unchanged.
- Always add a `%changelog` entry for every bump.

### Build steps

```bash
# Set up rpmbuild tree (once)
rpmdev-setuptree

# Set VERSION to match the Version: field in the target spec file
VERSION=1.20.25   # example — read from spec file, do not guess

# Commit all changes first — git archive only includes committed files
git archive --format=tar.gz --prefix=sudo-logger-${VERSION}/ HEAD \
    > ~/rpmbuild/SOURCES/sudo-logger-${VERSION}.tar.gz

# Build the RPMs
rpmbuild -ba rpm/sudo-logger-client.spec
rpmbuild -ba rpm/sudo-logger-server.spec
rpmbuild -ba rpm/sudo-logger-replay.spec

# RPMs land in:
ls ~/rpmbuild/RPMS/x86_64/
```

Use the `/rpm-builder` skill to be guided through the full versioning and
build workflow.

### What each RPM contains

| RPM | Binary | Config files | Systemd units | Other |
|---|---|---|---|---|
| `sudo-logger-client` | `sudo-logger-agent` → `/usr/bin/` | `/etc/sudo-logger/` (agent.conf, sandbox.yaml) | `sudo-logger-agent.service` | `sudo_logger_plugin.so` → `/usr/libexec/sudo/`; SELinux policy; 3 man pages; immutable audit rule; `refuse-stop.conf` drop-ins for sshd/sssd/rsyslog/polkit/chronyd/NetworkManager/firewalld/crond/gssproxy (no sudoers.d env_keep drop-in — see Chapter 11) |
| `sudo-logger-server` | `sudo-logserver` → `/usr/bin/` | `/etc/sudo-logger/` (cert dir, policy YAML) | `sudo-logserver.service`, `sudo-logserver-restart.timer` | man page; `sudologger` user/group |
| `sudo-logger-replay` | `sudo-replay-server` → `/usr/bin/` | `/etc/sudo-logger/` (auth YAML, risk-rules) | `sudo-replay.service` | Embedded React SPA (baked in binary); man page |

All three packages create the `sudologger` system user and group, install a
systemd preset that enables the service on install, and drop a
`refuse-stop.conf` drop-in to prevent accidental `systemctl stop` (use
`systemctl kill` during upgrades instead).

## Wire protocol

### Adding a new message type

When adding a new message type to the wire protocol, update all four layers
in this order:

**Step 1: Add the constant in `go/internal/protocol/protocol.go`**

```go
// go/internal/protocol/protocol.go
const (
    // ... existing constants ...
    MsgMyNewMessage = uint8(0xNN)   // direction: component→component; payload: description
)
```

Choose the next available hex value. Add a comment documenting the direction
and payload format.

**Step 2: Add a handler in `plugin.c` (if the plugin receives or sends it)**

In `plugin.c`, add a case to the relevant `switch (type)` block in
`monitor_thread_fn` (for messages received by the plugin) or call `send_msg`
with the new type constant (for messages sent by the plugin). Define the
matching `MSG_MY_NEW_MESSAGE` C constant to match the Go value.

**Step 3: Add a handler in `go/cmd/agent/plugin.go` (agent side)**

In `handlePluginConn`, add a case to the frame dispatch switch:

```go
case protocol.MsgMyNewMessage:
    // handle the message
```

If the message flows through the agent without processing, add it to the
proxy forwarding logic.

**Step 4: Add a handler in `go/cmd/server/handler.go` (if the server receives it)**

Add a case in `(srv *server) handleConn`'s per-connection frame dispatch loop.

**Step 5: Update this documentation**

Add the new message type to the table in Chapter 2 (Architecture, Wire
protocol section) and update the relevant sequence diagrams.

### Protocol constants (go/internal/protocol/protocol.go)

Message types now run through `0x1b`. This table was significantly out of
date (stopped at `0x14` and had one value wrong); verify against the source
directly if in doubt.

```go
const (
    MsgSessionStart             = uint8(0x01) // plugin→agent→server: JSON SessionStart
    MsgChunk                    = uint8(0x02) // plugin→agent→server: binary chunk
    MsgSessionEnd               = uint8(0x03) // plugin→agent→server: binary final_seq+exit_code
    MsgAck                      = uint8(0x04) // server→agent: binary seq+ts_ns+sig(64)
    MsgAckQuery                 = uint8(0x05) // plugin→agent: empty
    MsgAckResponse              = uint8(0x06) // agent→plugin: binary last_ack_ts_ns+last_seq
    MsgSessionReady             = uint8(0x07) // agent→plugin: empty — server OK, sudo may proceed
    MsgSessionError             = uint8(0x08) // agent→plugin: string error — sudo blocked
    MsgHeartbeat                = uint8(0x09) // agent→server: empty keepalive (every 400 ms)
    MsgHeartbeatAck             = uint8(0x0a) // server→agent: empty reply to heartbeat
    MsgServerReady              = uint8(0x0b) // server→agent: session accepted, agent may send SESSION_READY
    MsgSessionDenied            = uint8(0x0c) // server→agent AND agent→plugin: policy denial, string block message
    MsgFreezeTimeout            = uint8(0x0d) // agent→plugin: server unreachable too long, session will be terminated
    MsgSessionAbandon           = uint8(0x0e) // agent→server (new conn): freeze-timeout fired; payload = session_id UTF-8
    MsgSessionFreezing          = uint8(0x0f) // agent→server (new conn): session frozen due to network loss; payload = session_id UTF-8
    MsgDivergenceAlert          = uint8(0x10) // agent→server: JSON divergence metadata (DivergenceAlert)
    MsgSandboxAlert             = uint8(0x11) // agent→server: JSON sandbox violation (SandboxAlert)
    MsgFetchConfig              = uint8(0x12) // agent→server: UTF-8 config key (e.g. "sandbox.yaml")
    MsgConfigData               = uint8(0x13) // server→agent: UTF-8 YAML content (empty = not found)
    MsgSessionChallenge         = uint8(0x14) // server→agent→plugin: JSON SessionChallenge — justification required
    MsgSessionChallengeResponse = uint8(0x15) // plugin→agent→server: JSON SessionChallengeResponse — user's justification
    MsgSessionExpired           = uint8(0x16) // agent→plugin: approval window expired, session is being terminated
    MsgSessionWarning           = uint8(0x17) // agent→plugin: session will be terminated soon; payload = UTF-8 seconds left
    MsgSudoersSnapshot          = uint8(0x18) // agent→server: JSON SudoersSnapshot
    MsgSudoersError             = uint8(0x19) // agent→server: JSON SudoersError — failed to apply received config
    MsgHeartbeatAgent           = uint8(0x1a) // agent→server: periodic liveness signal; payload = UTF-8 host
    MsgResize                   = uint8(0x1b) // plugin→agent→server: binary ts_ns(8BE)+cols(2BE)+rows(2BE)
```

### API layer (ui/src/api/)

Each file in `api/` wraps a group of backend endpoints. All functions use
`apiFetch` from `client.ts`, which handles JSON serialisation and throws
`ApiError` (with `status` and `message` fields) on non-2xx responses.

| File | Endpoints wrapped |
|---|---|
| `client.ts` | Base `apiFetch` wrapper; `ApiError` class |
| `sessions.ts` | `GET /api/sessions`, `GET /api/session/events`, `DELETE /api/sessions/{tsid}` (note: plural, proxied to the log server's admin API when `--logserver-admin` is configured), `GET /api/session/cast` |
| `reports.ts` | `GET /api/report`, `GET /api/access-log` |
| `policy.ts` | `GET/PUT /api/risk-rules`, `GET/PUT /api/blocked-users`, `GET/PUT /api/whitelisted-users`, `GET /api/sudoers` |
| `config.ts` | `GET/PUT /api/siem-config`, `GET/PUT /api/auth-config`, `PUT /api/auth-mapping` (legacy `admin_groups` only), `GET/POST/DELETE /api/users`, `GET/POST/PUT/DELETE /api/roles`, `GET/PUT /api/retention`, `GET/PUT /api/sandbox`, `GET/PUT /api/jit-policy` |
| `approvals.ts` | `GET /api/approvals`, `POST /api/approvals/:id/approve`, `POST /api/approvals/:id/deny` |

### Adding a new UI page

1. **Create the view component** in `ui/src/views/MyPage.tsx`. Use TanStack
   Query's `useQuery` for data fetching and `useMutation` for writes.

2. **Add API functions** in the appropriate `ui/src/api/` file, or create a
   new file if it covers a new domain.

3. **Add TypeScript types** in `ui/src/types/` for any new API response shapes.

4. **Register the route** in `App.tsx`:
   ```tsx
   const MyPage = lazy(() => import('./views/MyPage'));
   // inside <Routes>:
   <Route path="/my-page" element={<MyPage />} />
   ```

5. **Add a navigation item** in `AppShell.tsx` using `NavLink`:
   ```tsx
   <NavLink to="/my-page">My Page</NavLink>
   ```

6. **Add a backend route** in `go/cmd/replay-server/main.go`:
   ```go
   mux.HandleFunc("/api/my-endpoint", s.handleMyEndpoint)
   ```
   Gate access with `can(r, store.PermXxx)` or `require(w, r, store.PermXxx)`.

### Running the dev server

```bash
cd go/cmd/replay-server/ui
npm run dev
```

Vite starts a dev server (default port 5173) with hot module replacement.
`/api` requests are proxied to `http://localhost:8080` (the replay server).
The replay server must be running separately:


## Pre-commit checks

Install pre-commit once:

```bash
pip install pre-commit
pre-commit install
```

Run all hooks manually:

```bash
pre-commit run --all-files
```

Hooks configured in `.pre-commit-config.yaml`:

| Hook ID | Tool | What it checks |
|---|---|---|
| `detect-secrets` | Yelp/detect-secrets v1.5.0 | Credentials and secrets in source files; uses `.secrets.baseline` |
| `go-vet` | `go vet` | Go static analysis: suspicious constructs |
| `golangci-lint` | golangci-lint (`--enable-only=govet,ineffassign,staticcheck`) | Go style, correctness, and performance lints |
| `go-test` | `go test -count=1 ./...` | All Go unit tests pass |
| `cppcheck` | cppcheck | C plugin: warnings (excluding missing system headers) |
| `flawfinder` | flawfinder | C plugin: security flaws at level 3 or higher |
| `trivy` | Trivy filesystem scan | Dependency and IaC vulnerability scanning (`go.mod`/`go.sum`/`Dockerfile`/`*.yaml`/`*.yml`) |

Plus the standard `pre-commit-hooks` set (`trailing-whitespace`, `end-of-file-fixer`,
`check-yaml`, `check-json`, `check-toml`, `check-merge-conflict`,
`check-added-large-files` at 1024 KB, `mixed-line-ending` fixed to LF).

The `go/vendor/` directory is excluded from all hooks (`exclude: ^go/vendor/`).

Fix all `HIGH` and `CRITICAL` severity findings from Trivy before committing.
`MEDIUM` and lower findings should be reviewed and suppressed with a `.trivyignore`
comment if they are not exploitable in this context.

## Code security guidelines

These rules apply to all contributors. They are not suggestions.

**TLS and authentication:**
- Never change `tls.RequireAndVerifyClientCert`, the minimum TLS version
  (`tls.VersionTLS13`), or the CA pool construction in `buildTLSConfig`
  without a security review. A weakening here breaks the audit integrity
  guarantee.
- Never remove mutual TLS from the agent-server connection. The wire protocol
  carries unredacted session data.
- Never change the token, OIDC, or proxy-header authentication logic in
  `replay-server/main.go` without considering the impact on audit access
  controls.

**File permissions and audit log integrity:**
- Never change the permissions on `/var/log/sudoreplay` or its subdirectories
  without reviewing the impact on log integrity. Session files must not be
  writable by the `sudologger` user after they are written.
- `DeleteSession` is the only supported deletion path. Direct filesystem
  deletion bypasses the audit log.

**Timestamps:**
- All timestamps displayed in the UI or included in generated documents
  must use the format `yyyy-mm-dd HH:mm` (24-hour clock), implemented in
  `ui/src/lib/date.ts` via `fmtDate`.

**Pre-release checklist:**
- Run `/security-review` before any release that touches TLS handling,
  authentication logic, file permissions, or the wire protocol.
- Run `pre-commit run --all-files` and fix all HIGH/CRITICAL Trivy findings
  before committing.

**Wire protocol integrity:**
- Never add a path that allows frames to be written to the log server
  connection without the agent's session ID scoping. The session ID is the
  only multiplexing key over the shared TLS connection.
- ACK signatures (ed25519) must not be skipped or weakened. They are the
  forensic proof that the server received each chunk.

**Deployment checklist after code changes:**

| Changed path | RPM to rebuild | Deploy to |
|---|---|---|
| `plugin/` or `go/cmd/agent/` | `sudo-logger-client` | All monitored hosts |
| `go/cmd/server/` | `sudo-logger-server` | Log server (gnarg) |
| `go/cmd/replay-server/` | `sudo-logger-replay` | Replay server (gnarg) |

After deploying, restart the relevant service:

```bash
sudo systemctl restart sudo-logger-agent   # on monitored hosts
sudo systemctl restart sudo-logger-server  # on log server
sudo systemctl restart sudo-logger-replay  # on replay server
```
