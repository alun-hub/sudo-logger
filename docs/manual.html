Welcome to the official sudo-logger documentation. This guide covers everything from system architecture to advanced API integration.

New in v1.24: Centralized Secret Redaction management and eBPF-based session auditing for su/pkexec.

Secret Redaction

sudo-logger features a multi-layer redaction engine that masks sensitive information locally on the monitored host before it is transmitted to the server.

1. Variable Assignment Masking

The engine automatically identifies variable assignments in the terminal stream (e.g. in .bashrc or .env files). If a variable name contains any of the following keywords, its value is masked:

KEY, SECRET, TOKEN, AUTH, PASS, PWD, PASSWORD, ACCESS

2. Entropy-Based Detection

For variables with unknown names (like KALLE=...), the engine performs entropy analysis. Any assignment where the value is a random-looking string of 24 characters or more (hex or base64) is automatically redacted.

3. Pattern Matching (Gitleaks)

Industry-standard patterns are built-in to detect specific secrets regardless of context:

  • Cloud: AWS Access Keys, Google API Keys.
  • SaaS: GitHub PATs, Stripe Secret Keys, Slack Webhooks.
  • Files: PEM-encoded private keys (SSH, SSL/TLS certificates).

Sudoers Management

The Replay UI provides a visual editor for /etc/sudoers.d/sudo-logger-managed. This allows administrators to manage access without risk of syntax errors locking out root access.

Snapshots & Recovery

Every time a policy is saved, a snapshot is created. You can view the history of changes and restore any previous version with a single click in the Policy → Sudoers tab.

eBPF Sandbox & Monitoring

sudo-logger uses eBPF (Extended Berkeley Packet Filter) to provide kernel-level security that cannot be bypassed by simply renaming binaries or using sub-shells.

Tracepoints

The agent attaches to the sys_enter_execve tracepoint. This allows us to:

  • Audit su/pkexec: Detect when a user tries to escape the sudo-logger plugin by starting a new privileged shell.
  • Enforce Policy: Instantly kill processes that match forbidden patterns in sandbox.yaml.
  • TTY Recording: Capture raw terminal input even if the user is running a binary that doesn't use standard TTY calls.
Performance: eBPF hooks run in the kernel. sudo-logger's hooks are optimized for low overhead (< 1ms latency per exec).

JIT (Just-In-Time) Workflows

Enforce "Four-Eyes" approval for high-risk commands. When a command matches a JIT policy, it is suspended until an administrator approves it.

The Lifecycle

  1. Trap: User runs a restricted command. Agent sends SIGSTOP to the process.
  2. Notify: A webhook is triggered (Slack/Mattermost) with a deep-link to the Replay UI.
  3. Review: An administrator reviews the request, host, and justification.
  4. Release: Upon approval, the agent sends SIGCONT and recording begins.

Security Architecture

The Replay Server is hardened against modern web attacks using a strict set of security headers.

Content Security Policy (CSP)

A restrictive CSP is enforced to prevent XSS and data exfiltration:

  • wasm-unsafe-eval: Required for the terminal player's high-performance WebAssembly engine.
  • frame-ancestors 'none': Prevents clickjacking by blocking the UI from being embedded in iframes.
  • connect-src 'self': Ensures the browser only communicates with the Replay API.

Isolation Headers

  • Cross-Origin-Opener-Policy (COOP): Set to same-origin to isolate the execution environment.
  • Strict-Transport-Security (HSTS): Enforced for 2 years (63072000 seconds) including subdomains.
