Complete reference for installation, configuration, API usage, and operations.
v1.20+ Go 1.25+ RHEL / Fedora / Rocky sudo 1.9.13+

Introduction

sudo-logger records every sudo session — terminal I/O, metadata, and risk context — and streams it to a central audit server in real time. Recordings survive agent failures via a freeze-and-buffer mechanism. Sessions are searchable, replayable, and scored against a configurable risk rule set.

Three core components must be deployed together:

  • Plugin (plugin.so) — loaded by sudo on every monitored host. Captures I/O and pushes events to the local agent via a Unix socket.
  • Agent (sudo-logger-agent) — local daemon that buffers and forwards session data to the log server over mTLS. Applies the sandbox policy and freeze-on-disconnect logic.
  • Log server (sudo-logger-server) — central receiver. Writes sessions to disk (or S3). Manages JIT approvals and SIEM forwarding.
  • Replay server (sudo-logger-replay) — web UI. Reads sessions from the same storage backend, scores them against risk rules, and provides the management API.

Architecture

Monitored host Central server (gnarg / k3s) ───────────────────────────── ───────────────────────────────────── ┌─────────────────────────────────┐ sudo ──plugin.so──► agent.sock │ sudo-logger-server :9876 │ │ │ mTLS listener │ agent (Go) │ ├─ /api/session/* (ingest) │ ├─ buffer │ ├─ /api/approvals (JIT REST) │ ├─ freeze/thaw │ └─ /healthz + /metrics :9877 │ ├─ sandbox BPF └────────┬────────────────────────┘ └─ HMAC-signed chunks │ shared filesystem or S3 │ mTLS │ └──────────────────►│ /var/log/sudoreplay/ │ ┌───────┴────────────────────────┐ │ sudo-logger-replay :8080 │ │ ├─ /api/* (REST, RBAC) │ │ ├─ /metrics │ │ └─ / (React SPA) │ └────────────────────────────────┘ │ Browser (OIDC / Basic Auth)

Sessions are written as a directory tree under logdir: YYYY/MM/DD/<sessionid>/, containing a JSON metadata file and optional binary I/O chunks. The replay server embeds the React SPA and serves it from the same binary.

RPM Install

Build RPMs with the /rpm-builder script (runs in a Fedora 42 container), then deploy:

# Build — run on dev machine
/rpm-builder client   # produces sudo-logger-client-*.rpm
/rpm-builder server   # produces sudo-logger-server-*.rpm
/rpm-builder replay   # produces sudo-logger-replay-*.rpm

# Deploy client to all monitored hosts
sudo dnf install sudo-logger-client-*.rpm
sudo systemctl enable --now sudo-logger-agent

# Deploy server + replay to gnarg
sudo dnf install sudo-logger-server-*.rpm sudo-logger-replay-*.rpm
sudo systemctl enable --now sudo-logger-server sudo-logger-replay
File layout after RPM install
/usr/libexec/sudo/plugin.so — sudo I/O plugin
/usr/bin/sudo-logger-agent — agent binary
/etc/sudo-logger/ — all config files
/var/log/sudoreplay/ — session storage
/etc/sudoers.d/sudo-logger — Plugin line for sudo

Manual Install

# Build all binaries
make -C go

# Install plugin
sudo install -m 0755 plugin/plugin.so /usr/libexec/sudo/sudo_logger.so
echo 'Plugin sudo_logger /usr/libexec/sudo/sudo_logger.so' \
  | sudo tee /etc/sudo.conf.d/sudo-logger.conf

# Install agent
sudo install -m 0755 go/cmd/agent/agent /usr/bin/sudo-logger-agent
sudo install -m 0644 go/systemd/sudo-logger-agent.service \
  /etc/systemd/system/sudo-logger-agent.service
sudo systemctl daemon-reload
sudo systemctl enable --now sudo-logger-agent

Verification

# Check agent is connected to log server
sudo journalctl -u sudo-logger-agent -n 30

# Trigger a test session
sudo -u nobody id

# Confirm session arrived at replay server
curl -s http://localhost:8080/api/sessions | python3 -m json.tool | head -40

File integrity can be verified at any time: ./scripts/verify-integrity.sh <session-dir>

Agent Configuration (agent.conf)

The agent reads /etc/sudo-logger/agent.conf (INI-style, key = value, # comments). All keys are optional — defaults are production-safe.

KeyDefaultDescription
server logserver:9876 Log server address (host:port). Uses mTLS — cert/key/ca must match.
socket /run/sudo-logger/plugin.sock Unix socket path shared with the sudo plugin. Must be writable by sudo (root-owned, mode 0700).
cert /etc/sudo-logger/client.crt Client TLS certificate for connecting to the log server. Presented during mTLS handshake.
key /etc/sudo-logger/client.key Private key for the client TLS certificate. File must be mode 0600, root-readable only.
ca /etc/sudo-logger/ca.crt CA certificate used to verify the log server's TLS certificate.
verify_key /etc/sudo-logger/ack-sign.pub ed25519 public key for verifying ACK signatures from the log server. Prevents replay attacks on the chunk acknowledge flow.
mask_pattern empty Regex applied to terminal output. Matching lines are replaced with [REDACTED] before storage. Use to suppress passwords in non-PTY sessions.
freeze_timeout 3m How long to freeze new sudo sessions when the log server is unreachable. Format: Go duration (30s, 5m, 1h). After this period, sessions are allowed to proceed without blocking.
idle_timeout 30m Inactivity timeout for the agent's log server connection. Connection is re-established automatically.
debug false Enable verbose logging to journald. Include raw chunk sizes and TLS handshake detail. Never enable in production — logs may contain session data.
disclaimer empty Banner message shown to users before each sudo session. Leave empty to disable. Example: This system is monitored. All sudo activity is recorded.
disclaimer_color yellow ANSI color for the disclaimer banner. Options: red, yellow, green, blue, white.
ebpf true Enable the eBPF TTY recorder for sessions not handled by the sudo plugin (e.g. su, screen, tmux, pkexec). Requires kernel 5.15+ and CAP_BPF.
sandbox_config /etc/sudo-logger/sandbox.yaml Path to the sandbox policy file. The agent fetches an updated copy from the log server every 60 s — this path is only used as a read-fallback when the server is unreachable.
hostname OS hostname Override the hostname reported to the log server. Useful in containerised environments or when the OS hostname is not meaningful.

Example agent.conf

# /etc/sudo-logger/agent.conf
server          = logserver.internal:9876
cert            = /etc/sudo-logger/client.crt
key             = /etc/sudo-logger/client.key
ca              = /etc/sudo-logger/ca.crt
verify_key      = /etc/sudo-logger/ack-sign.pub
freeze_timeout  = 3m
idle_timeout    = 30m
ebpf            = true
disclaimer      = This system is monitored. All sudo activity is recorded and reviewed.
disclaimer_color = yellow

Risk Rules YAML Reference

File: /etc/sudo-logger/risk-rules.yaml — loaded by the replay server and recalculated per session on read. Each rule adds its score to the session's total (capped at 100). Rules are evaluated independently and scores accumulate.

Rule structure

FieldTypeDescription
scoreint (1–100)Points added when this rule matches. Multiple matching rules accumulate.
idstringUnique identifier. Use snake_case. Shown in anomaly reports.
reasonstringShort label shown in the session info bar and anomalies tab. Be specific.
command.contains_any[]stringCommand line must contain at least one of these substrings (case-insensitive).
command.also_any[]stringAND — command line must also contain at least one of these.
command_base_any[]stringMatches only the program basename (argv[0]). More reliable for shell detection than substring.
content.contains_any[]stringScans tty output for at least one substring. ANSI escape codes are stripped. Only available for sessions with recorded I/O.
content.also_any[]stringAND — tty output must also contain at least one of these.
runasstringOnly match sessions where sudo ran as this user. Default (empty) matches any.
min_duration_sintMinimum session duration in seconds. Useful for flagging long interactive shells.
incompleteboolOnly match sessions that ended without a clean SESSION_END event.
after_hoursboolOnly match sessions that started between 23:00 and 05:59 local time.

Score thresholds

Score rangeLevelColour
0–24LowGray
25–49MediumYellow
50–74HighOrange
75–100CriticalRed

Complete example

rules:
  # Stop or disable the audit daemon
  - score: 80
    id:    stop_auditd
    reason: Audit daemon stopped or disabled
    command:
      contains_any: [auditd]
      also_any:     [stop, disable, mask, kill]

  # Direct root shell via any interpreter
  - score: 50
    id:    root_shell
    reason: Direct root shell
    command_base_any: [bash, sh, zsh, fish, ksh, dash]

  # Auth file manipulation inside a shell
  - score: 70
    id:    auth_file_tamper
    reason: Auth file manipulation
    content:
      contains_any: [/etc/passwd, /etc/shadow, /etc/sudoers, /etc/pam.d]

  # Late-night root activity
  - score: 20
    id:    after_hours_root
    reason: After-hours root activity
    runas: root
    after_hours: true

  # Long-running interactive shell (>10 min)
  - score: 30
    id:    long_shell
    reason: Long interactive root shell (>10 min)
    command_base_any: [bash, sh, zsh]
    min_duration_s: 600

  # Incomplete session — agent was killed
  - score: 40
    id:    incomplete_session
    reason: Session ended unexpectedly (agent killed?)
    incomplete: true

  # Kernel module loading
  - score: 90
    id:    kernel_module
    reason: Kernel module loaded
    command:
      contains_any: [insmod, modprobe, rmmod]

  # SELinux disabled
  - score: 95
    id:    selinux_disabled
    reason: SELinux enforcement changed
    command:
      contains_any: [setenforce, semodule]

Sandbox Policy YAML Reference

File: /etc/sudo-logger/sandbox.yaml — served to agents on demand. The agent applies it via eBPF seccomp + BPF-LSM. Changes propagate to all agents within 60 seconds without restart.

Top-level sections

SectionDescription
featuresBoolean flags controlling which BPF enforcement modules are active.
protectLists of resources that the sandbox will deny access to for sudo sessions.

features keys

KeyDefaultDescription
deny_netlinktrueBlock AF_NETLINK socket creation. Prevents iptables, nftables, ip route, auditctl from working inside the session.
deny_mounttrueBlock bind-mount operations. Prevents overmounting protected paths.
deny_ptracetrueBlock ptrace. Prevents code injection into root daemons from within the session.
deny_cap_audit_controltrueDrop CAP_AUDIT_CONTROL. Defence-in-depth for auditctl bypass.
deny_cap_net_adminfalseDrop CAP_NET_ADMIN. Blocks raw sockets and network configuration. May break iproute2.
deny_cap_sys_moduletrueDrop CAP_SYS_MODULE. Prevents kernel module loading via insmod/modprobe.
deny_cap_mac_admintrueDrop CAP_MAC_ADMIN. Prevents setenforce and semodule from within the session.
deny_cap_sys_rawiotrueDrop CAP_SYS_RAWIO. Blocks direct access to /dev/mem, /dev/port.
deny_cap_sys_boottrueDrop CAP_SYS_BOOT. Prevents kexec_load and reboot from within the session.
deny_systemd_ipcfalseRestrict systemd D-Bus IPC. Blocks systemctl, loginctl from within sessions. Off by default because many legitimate admin tasks need it.

protect sub-keys

KeyDescription
filesPaths (files or directories) that sessions may not write to or delete.
forbiddenExecutable paths that sessions may not exec. Checked against execve path.
noexecDirectories where execution is blocked (no exec from /tmp, /dev/shm etc.).
devicesDevice nodes that sessions may not open.
procSpecific /proc entries that sessions may not write to.
socketsUnix sockets that sessions may not connect to.
processesProcess comm names that sessions may not send signals to.

Complete example

  deny_mount:             true
  deny_ptrace:            true
  deny_cap_audit_control: true
  deny_cap_net_admin:     false

  deny_cap_mac_admin:     true
  deny_cap_sys_rawio:     true

    - /etc/sudoers
    - /etc/sudoers.d
    - /etc/sudo-logger

    - /etc/rsyslog.conf
    - /etc/systemd
    - /root/.ssh
    - /etc/pki
  forbidden:
    - /usr/bin/systemd-run
    - /usr/sbin/auditctl
    - /usr/sbin/setenforce
    - /usr/sbin/semodule
    - /usr/sbin/kexec
    - /usr/bin/busctl

    - /tmp
    - /var/tmp
    - /dev/shm

    - /dev/kmem

    - /proc/kcore

Compatibility note
The sandbox uses eBPF BPF-LSM and requires kernel 5.15+ with CONFIG_BPF_LSM=y. On Fedora 38+, Rocky Linux 9+ and RHEL 9+ all prerequisites are met. Disable with ebpf = false in agent.conf on older kernels.

OPA JIT Policy

The OPA JIT (Just-In-Time) policy controls what happens when a user runs sudo. Depending on the matching rule, the session is either allowed through immediately, held for manual approval (challenge), or denied outright. The policy is authored as YAML in the UI and compiled to an OPA Rego module by the log server.

Navigate to Policy → JIT in the replay server to manage this policy. The view has three tabs: Rules, Groups, and Compiled Rego.

Approval Workflow

The full end-to-end flow when a user runs sudo:

User runs: sudo bash agent (on monitored host) │ ├─ 1. Prompts user: "Enter justification for this sudo session:" │ (user types a reason, or presses Enter if optional) │ ├─ 2. Sends approval request to log server: │ { user, host, command, justification, timestamp } │ ├─ 3. Log server evaluates OPA policy: │ ┌─────────────────────────────────────┐ │ │ action = allow → proceed now │ │ │ action = deny → reject now │ │ │ action = challenge → wait for human│ │ └─────────────────────────────────────┘ │ ├─ 4a. ALLOW → agent receives approval, sudo proceeds. │ ├─ 4b. DENY → agent receives rejection, sudo exits with error. │ User sees the deny reason on their terminal. │ └─ 4c. CHALLENGE → agent blocks sudo (terminal shows "Waiting for approval…") │ ├─ Approver opens replay UI → Approvals tab ├─ Sees pending request: user / host / command / justification ├─ Clicks Approve or Deny (with optional reason) │ └─ Agent receives decision → sudo proceeds or exits
Timeout behaviour
If no approver responds within the configured timeout (default: 5 minutes), the agent falls back to the default_action set in the policy. Set a short timeout for high-security environments. Configure the timeout in Config → JIT or via -approval-policy.
Prerequisites
JIT requires the log server to be started with -approval-token and -approval-policy flags. The replay server needs -logserver-admin and -logserver-admin-token. The Approvals tab only appears in the UI when these are configured. Users in the Whitelisted Users list (Policy → Access Control) skip the JIT prompt entirely.

Actions

Each rule has an action — the outcome applied when the rule's conditions all match. There are three actions:

ActionBadge colourWhat happens
CHALLENGE Blue The session is held. An approver must explicitly approve it in the Approvals tab before sudo proceeds. The user sees "Waiting for approval…" on their terminal. Use for commands or users that require oversight but are not outright forbidden.
ALLOW Green The session is automatically approved. No human approval required. The user's justification is still recorded for audit purposes. Use for trusted users, service accounts, or during business hours for known-good users.
DENY Red The session is rejected immediately. The user sees the deny reason on their terminal and sudo exits with an error. No approval can override a deny. Use to absolutely block specific commands or users.

Default action

The "Default action if no rules match" dropdown at the top of the Rules tab controls what happens when no rule matches the incoming request. Options:

  • Challenge (ask why) — holds every unmatched session for approval. Safest default: nothing slips through without oversight.
  • Allow (no prompt) — automatically approves every unmatched session. Use only when you have explicit deny rules covering all dangerous operations.
Rule order matters for DENY
Rules are evaluated top to bottom. The first matching deny rule wins immediately. For allow and challenge, all matching rules are collected and the most restrictive wins (deny > challenge > allow). Place broad allow rules below narrow deny rules.

Conditions

A rule fires only when all its conditions are satisfied simultaneously (AND logic). An empty condition field is a wildcard — it matches everything. In the UI the Conditions column shows a compact summary: users: * hosts: * commands: * means "no restrictions on user, host, or command".

FieldTypeDisplay in UIDescription
users []string users: alice, @sre Match if the sudo user equals one of these values. Supports glob patterns (deploy-*) and @groupname references. Empty = all users (*).
hosts []string hosts: prod-* Match if the session host equals one of these values. Supports globs. Empty = all hosts (*).
commands []string commands: bash, sh Match if the command basename (argv[0]) is one of these. Globs supported. Empty = all commands (*).
runas []string runas: root Match if sudo -u <user> targets one of these users. Empty = any runas user.
sys_groups []string groups: wheel Match if the user belongs to one of these system groups (resolved live via NSS/LDAP). Useful when your user directory is the source of truth for role membership.
weekdays []int weekdays: 1-5 Match if today is one of these weekdays. 0 = Sunday, 1 = Monday … 6 = Saturday. Empty = any day.
hour_from / hour_to int (0–23) time: 20:00–6:00 Match if the current hour is within this window. The UI shows it as time: HH:00–HH:00. Wrap-around is supported — hour_from: 20, hour_to: 6 matches 20:00–05:59. Both must be set to activate the time filter; -1 means "no restriction".

Time-window example

The screenshot shows a rule with time: 20:00–6:00. This is an ALLOW rule that fires for "SRE on-call" during night hours — everyone else (or daytime sessions) falls through to the default CHALLENGE action. In YAML this looks like:

rules:
  # Night-time on-call: allow without approval (20:00 to 06:00)
  - id:      oncall_night
    comment: If SRE oncall
    action:  allow
    hour_from: 20
    hour_to:   6

  # Default catch-all: challenge everything else
  - id:      default_challenge
    comment: Default
    action:  challenge
    # (no other conditions = matches all users, hosts, commands)
Tip — using the default action instead of an explicit catch-all rule
The two rules above can be simplified: just set the Default action dropdown to Challenge and keep only the oncall rule. An explicit catch-all is only needed when you want a different comment to appear in the approvals log.

Groups Tab

The Groups tab lets you define named sets of users, hosts, or any other string values. Once defined, reference a group anywhere in a rule's condition fields by prefixing the name with @.

groups:
  sre:
    - alice
    - bob
    - carol
  prod-hosts:
    - prod-web-*
    - prod-db-*
    - prod-cache-*
  dangerous-commands:
    - bash
    - sh
    - python3
    - perl

Use these groups in rules:

rules:
  # SRE can run dangerous commands on prod — but must give a reason
  - id:      sre_prod_shell
    comment: SRE shell on production
    action:  challenge
    users:   [@sre]
    hosts:   [@prod-hosts]
    commands: [@dangerous-commands]

  # Non-SRE users running a shell on prod: deny
  - id:      block_prod_shell
    comment: Non-SRE shell on production is blocked
    action:  deny
    hosts:   [@prod-hosts]
    commands: [@dangerous-commands]
System groups vs. JIT groups
JIT groups (defined here in the Groups tab) are maintained in the policy YAML. Use them when you need an exact list independent of your directory service.

System groups (sys_groups condition field) are resolved live from the OS (/etc/group, LDAP, or AD). Use them when your user directory is already the authoritative source of role membership. System groups do not need to be defined in the Groups tab.

Compiled Rego Tab

The Compiled Rego tab shows the full OPA Rego module that the log server evaluates for every approval request. The left panel is an optional text area for hand-written Rego additions; the right panel (read-only) shows the complete compiled module.

When to use custom Rego

The YAML rule builder covers most use cases. Write custom Rego when you need logic that the YAML schema cannot express:

  • Cross-field conditions (e.g. "only if user AND runas AND command all match AND it's not a Friday")
  • Calls to external data sources via OPA's http.send
  • Complex string operations (regex, substring checks on justification text)
  • Overriding the final decision variable directly

Rego extension points

The compiled module exposes two set rules you can extend from the custom Rego panel:

# Append to the allow set — these sessions bypass challenge
_any_allow if {
    input.user == "emergency-break-glass"
}

# Append to the deny set — these sessions are blocked regardless
_any_deny if {
    input.command == "visudo"
    not input.user == "root"
}

Custom Rego is appended after the compiled rules, so it can reference compiled rule sets and override the final decision output variable.

Full YAML Reference

Top-level fields

FieldTypeDescription
default_actionstringallow or challenge. Applied when no rules match. Default: challenge.
groupsmap[string][]stringNamed groups. Reference as @name in rule condition fields.
rules[]RuleOrdered list of policy rules. All matching rules are collected; deny beats challenge beats allow.
raw_regostringOptional hand-written Rego appended to the compiled module.

Rule fields

FieldTypeRequiredDescription
actionstringYesallow, challenge, or deny.
idstringNoUnique identifier, shown in decision logs and the UI. Use snake_case.
commentstringNoHuman-readable note visible in the UI Comment column.
users[]stringNoMatch on sudo user. Globs and @group allowed. Empty = all.
hosts[]stringNoMatch on hostname. Globs and @group allowed. Empty = all.
commands[]stringNoMatch on command basename. Globs and @group allowed. Empty = all.
runas[]stringNoMatch on the target user (sudo -u). Empty = all.
sys_groups[]stringNoMatch if the user is a member of any of these OS groups (live NSS lookup).
weekdays[]intNo0=Sun … 6=Sat. Empty = all days.
hour_fromintNoStart of time window (inclusive, 0–23). Use -1 to disable. Shown in UI as time: HH:00–…
hour_tointNoEnd of time window (exclusive, 0–23). Wrap-around supported (e.g. 20→6 = night). Use -1 to disable.

Complete production example

# /etc/sudo-logger/approval-policy.yaml
default_action: challenge

groups:
  sre:
    - alice
    - bob
    - carol
  infra-hosts:
    - prod-web-*
    - prod-db-*
  dangerous:
    - bash
    - sh
    - zsh
    - python3

rules:
  # Hard deny: nobody may change audit configuration
  - id:      no_audit_changes
    comment: Audit configuration changes are blocked
    action:  deny
    commands: [auditctl, ausearch, aureport, setenforce]

  # SRE on production: allow without approval during business hours
  - id:      sre_business_hours
    comment: SRE on prod 08:00-18:00 weekdays
    action:  allow
    users:   [@sre]
    hosts:   [@infra-hosts]
    weekdays: [1, 2, 3, 4, 5]
    hour_from: 8
    hour_to:   18

  # SRE on-call: allow at night (20:00–06:00)
  - id:      sre_oncall
    comment: If SRE oncall
    action:  allow
    users:   [@sre]
    hour_from: 20
    hour_to:   6

  # Service accounts: always allow (CI/CD, monitoring)
  - id:      service_accounts
    comment: Service account — no approval needed
    action:  allow
    users:   [deploy-bot, prometheus, nagios]

  # Everyone else: challenge (requires human approval + justification)
  # (handled by default_action: challenge above — no rule needed)

SIEM Forwarding YAML Reference

File: /etc/sudo-logger/siem.yaml — read by both the log server and replay server. The log server does the actual forwarding; the replay server provides the UI to manage the config.

Top-level fields

FieldTypeDescription
enabledboolMaster switch. Set to false to disable all forwarding without deleting the config.
transportstringhttps, syslog, or stdout.
formatstringjson (default), cef (ArcSight), or ocsf (OCSF 1.0).
replay_url_basestringBase URL of this replay server, e.g. https://replay.example.com. Appended to SIEM events as a deep-link to the session.
httpsobjectHTTPS transport configuration (ignored unless transport: https).
syslogobjectSyslog transport configuration (ignored unless transport: syslog).

https object

FieldDescription
urlFull HTTPS endpoint URL. Example: https://splunk.example.com:8088/services/collector/event
tokenBearer token or Splunk HEC token. Sent as Authorization: Splunk <token> or Bearer <token>.
tls.caPath to CA certificate for verifying the SIEM endpoint's TLS certificate.
tls.certPath to client certificate for mTLS to the SIEM endpoint.
tls.keyPath to client private key for mTLS.

syslog object

FieldDescription
addrSyslog server address. Example: syslog.example.com:514
protocoludp, tcp, or tcp-tls.
tls.caCA certificate for verifying the syslog server (tcp-tls only).
tls.certClient certificate for mTLS to syslog server (tcp-tls only).
tls.keyClient private key for mTLS to syslog server (tcp-tls only).

    key:  /etc/sudo-logger/siem-client.key
Data sovereignty
All session events including sensitive terminal I/O are forwarded in real time. Ensure your SIEM endpoint is protected by high-entropy tokens and robust TLS. Events are buffered locally on the log server for up to 1 hour if the SIEM is unreachable.

The replay server uses RBAC with fine-grained permissions. Roles are defined in the UI (Config → Users & Roles) and assigned to users. The built-in admin role has all permissions and is assigned to users listed in -admin-users.

Authentication Modes

ModeFlagNotes
HTTP Basic Auth -htpasswd /path/to/.htpasswd Bcrypt hashes only (htpasswd -B). Reloaded on SIGHUP. Use with TLS — Basic Auth over plain HTTP is insecure.
Trusted header -trusted-user-header X-Forwarded-User For use behind oauth2-proxy or a reverse proxy that sets the header after authentication. Only trusted from 127.0.0.1.

Blocked Users

Managed at Policy → Access Control. The log server reloads the blocked users file every 30 seconds. When a user is blocked, the agent shows the configured block message and prevents the session from starting.

# /etc/sudo-logger/blocked-users.yaml
config:
  block_message: "Your sudo access has been suspended. Contact your manager."
  users:
    - username: jsmith
      hosts: [prod-web-01, prod-web-02]
      reason: "Security incident investigation (ticket #4192)"
      blocked_at: 1749823200
    - username: malicious
      hosts: []   # empty = all hosts
      reason: "Account compromised"
      blocked_at: 1749823200

Whitelisted Users

Users in the whitelist bypass the JIT approval prompt entirely — their sudo sessions proceed without requiring justification or approval. Managed at Policy → Access Control.

# /etc/sudo-logger/whitelisted-users.yaml
config:
  users:
    - username: alice
      hosts: [prod-db-01]
      reason: "DBA — permanent exception approved by CISO"
    - username: deploy-bot
      hosts: []   # empty = all hosts
      reason: "CI/CD service account"

Backup & Rotation

Session files are written to logdir/YYYY/MM/DD/<id>/. The directory structure makes daily snapshots trivial.

Daily backup (local storage)

# /etc/cron.daily/sudo-logger-backup
#!/bin/bash
set -euo pipefail
LOGDIR=/var/log/sudoreplay
BACKUP=/mnt/backup/sudo-logger
DATE=$(date +%Y-%m-%d)

# Snapshot previous day to backup storage (rsync + hardlinks)
rsync -a --link-dest="$BACKUP/latest" \
  "$LOGDIR/$(date -d yesterday +%Y/%m/%d)/" \
  "$BACKUP/$DATE/"

# Update 'latest' symlink
ln -sfn "$BACKUP/$DATE" "$BACKUP/latest"

# Verify integrity of backed-up sessions
find "$BACKUP/$DATE" -name '*.json' -exec \
  /opt/sudo-logger/scripts/verify-integrity.sh {} \;

Upgrade Procedure

  1. Build the new RPMs: /rpm-builder client|server|replay
  2. On the log server, install and restart in order: server first, then replay. This prevents the replay server from reading sessions that the new server format hasn't yet created.
  3. On monitored hosts, install the new client package and restart the agent. Sessions in progress are completed with the old agent before drain.
  4. Check journalctl -u sudo-logger-server -n 50 and journalctl -u sudo-logger-agent -n 50 for errors after restart.
  5. Run a test session: sudo -u nobody id and confirm it appears in the replay UI.
Database migrations
The replay server applies PostgreSQL migrations automatically at startup (distributed storage only). Always back up the database before upgrading. The migration version is visible in the startup logs.

TLS Setup (Internal CA)

# Generate CA
openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:4096 -out ca.key
openssl req -new -x509 -key ca.key -days 3650 \
  -subj "/CN=sudo-logger CA" -out ca.crt

# Generate server certificate (for log server)
openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:2048 -out server.key
openssl req -new -key server.key \
  -subj "/CN=logserver.internal" -out server.csr
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -days 365 -out server.crt

# Generate client certificate (for agent — per host)
HOSTNAME="prod-web-01"
openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:2048 \
  -out "client-${HOSTNAME}.key"
openssl req -new -key "client-${HOSTNAME}.key" \
  -subj "/CN=${HOSTNAME}" -out "client-${HOSTNAME}.csr"
openssl x509 -req -in "client-${HOSTNAME}.csr" -CA ca.crt -CAkey ca.key \
  -CAcreateserial -days 365 -out "client-${HOSTNAME}.crt"

# Generate ACK signing key
openssl genpkey -algorithm ed25519 -out ack-sign.key
openssl pkey -in ack-sign.key -pubout -out ack-sign.pub

# Distribute to agents
scp ca.crt ack-sign.pub "client-${HOSTNAME}.crt" "client-${HOSTNAME}.key" \
  "root@${HOSTNAME}:/etc/sudo-logger/"

Troubleshooting — Agent

Agent can't connect to log server

sudo journalctl -u sudo-logger-agent -n 50
# Look for: "tls: certificate signed by unknown authority"
# or: "connection refused"

# Check cert paths
sudo openssl verify -CAfile /etc/sudo-logger/ca.crt \
  /etc/sudo-logger/client.crt

# Test raw TLS connection
openssl s_client -connect logserver:9876 \
  -cert /etc/sudo-logger/client.crt \
  -key /etc/sudo-logger/client.key \
  -CAfile /etc/sudo-logger/ca.crt

Sessions are freezing

Symptoms: sudo hangs at startup for up to freeze_timeout seconds. Root cause: agent cannot reach the log server.

# Check agent status
systemctl status sudo-logger-agent

# Verify network connectivity to log server
nc -zv logserver.internal 9876

# Reduce freeze timeout temporarily for testing
# Edit /etc/sudo-logger/agent.conf:
# freeze_timeout = 5s
systemctl restart sudo-logger-agent

"chunk data truncated" or "session incomplete"

Indicates the plugin socket had a write error mid-session. Usually caused by the agent being restarted while sessions were in progress. Check: journalctl -u sudo-logger-agent --since "1 hour ago" for restart events.

eBPF not loading

# Check kernel version (requires 5.15+)
uname -r

# Check BPF LSM is enabled
cat /sys/kernel/security/lsm | grep -o bpf

# Check CAP_BPF is available
sudo /usr/bin/sudo-logger-agent --check-bpf

# Disable eBPF if not needed:
# ebpf = false in /etc/sudo-logger/agent.conf

Troubleshooting — Log Server

Approvals API returns 503

The approval API is disabled when the log server is started without -approval-token. Check: grep approval-token /etc/systemd/system/sudo-logger-server.service.d/*.conf

Sessions not appearing

# Check log server is receiving connections
sudo journalctl -u sudo-logger-server -n 50 | grep "new session"

# Check disk space on logdir
df -h /var/log/sudoreplay

# Check file permissions
ls -la /var/log/sudoreplay/
stat /var/log/sudoreplay

SIEM forwarding not working

# Test HTTPS endpoint manually
curl -v -H "Authorization: Splunk <token>" \
  -H "Content-Type: application/json" \
  -d '{"event": "test"}' \
  https://splunk.example.com:8088/services/collector/event

# Check log server logs for SIEM errors
sudo journalctl -u sudo-logger-server -n 100 | grep -i siem

# Reload SIEM config (server reads it every 5 min; or restart to force reload)
sudo systemctl reload sudo-logger-server  # sends SIGHUP

Troubleshooting — Replay Server

Blank page / React app not loading

Old session metadata files may not have the started_at field in ISO 8601 format. The UI handles both Unix timestamps (numbers) and ISO strings — if you see — dates, check the raw metadata JSON:

cat /var/log/sudoreplay/2026/06/11/<session-id>/meta.json | python3 -m json.tool

grep admin-users /etc/systemd/system/sudo-logger-replay.service.d/*.conf

# Add username to -admin-users flag and restart

Approvals tab is missing

The Approvals tab only appears when -logserver-admin is set on the replay server. Verify the flag is present and the log server's health port is reachable from the replay server:

curl http://logserver.internal:9877/healthz

Troubleshooting — Plugin

Plugin not loading

# Verify the Plugin line in sudo.conf
grep Plugin /etc/sudo.conf /etc/sudo.conf.d/*.conf 2>/dev/null

# Check plugin file exists and is executable
ls -la /usr/libexec/sudo/sudo_logger.so

# Test with sudo -V
sudo -V 2>&1 | grep -i plugin

# Check SELinux context
ls -lZ /usr/libexec/sudo/sudo_logger.so
# Expected: system_u:object_r:sudo_exec_t:s0
# Restore if wrong:
restorecon -v /usr/libexec/sudo/sudo_logger.so

Plugin socket errors

# Check socket exists
ls -la /run/sudo-logger/plugin.sock

# Check agent is running
systemctl status sudo-logger-agent

# Restart agent to recreate socket
systemctl restart sudo-logger-agent

Sessions captured by eBPF but not plugin

If you see sessions with the ebpf badge but no sessions from the plugin, the plugin is either not loaded or not writing to the socket. Check the agent log for "no plugin" warnings. Also verify /etc/sudo.conf has the correct Plugin line and the sudo version is 1.9.13+.

Still stuck?
Enable debug = true in agent.conf, restart the agent, reproduce the issue, and check journalctl -u sudo-logger-agent -n 200 for detailed output. Remember to disable debug after collecting logs — it may capture session data.
