Executive Summary

sudo-logger is the industry's most advanced audit recording solution for Linux privileged access. Traditional systems capture only the execution event, but sudo-logger records the entire interactive session as a high-definition, cryptographically signed stream.

Designed for the Zero-Trust Enterprise, the system ensures absolute Chain of Custody by evacuating audit data from monitored hosts in sub-second real-time. Even a root user cannot alter their actions once recorded, as the data no longer resides on the local machine.

Enterprise Grade: Scales to tens of thousands of nodes, supports S3-compatible object storage, PostgreSQL metadata, and full OIDC single sign-on.

How it works

User runs sudo │ ▼ ┌─────────────────────┐ │ sudo C plugin │ Loaded by sudo for every invocation. │ (plugin.so) │ Records stdin/stdout/tty I/O. │ │ Blocks sudo entirely if agent is unavailable. │ │ Freezes child process if ACKs go stale. └────────┬────────────┘ │ Unix socket (/run/sudo-logger/plugin.sock) ▼ ┌─────────────────────┐ │ sudo-logger-agent │ Local daemon running as root. │ (Go) │ Bridges plugin ↔ server. │ │ eBPF: records SSH/TTY logins & su/pkexec. │ │ Tracks ACK state per session. └────────┬────────────┘ │ Mutual TLS (TCP 9876) ▼ ┌─────────────────────┐ │ sudo-logserver │ Central log server. │ (Go) │ Receives session data. │ │ Sends ed25519-signed ACKs. └─────────────────────┘

The system maintains a tight ACK loop. If ACKs and heartbeats stop arriving (network loss, server crash), the child process is frozen within ~800 ms — no further input reaches it until ACKs resume or the session is killed.

Architecture & Components

1. C Plugin (plugin.so)

Implements the sudo_plugin.h API. Hooks into the terminal I/O loop of the sudo binary. By operating at the TTY level, it captures data before it can be obfuscated by sub-shells or interactive editors like vim.

  • Zero Buffer: All I/O is forwarded immediately to the agent.
  • Monitor Thread: A background thread polls ACK state every 150ms.
  • Fail-Closed: Terminate the active sudo session if the agent socket drops mid-session.

2. Agent (Go)

The host-level security orchestrator (sudo-logger-agent). It consolidates four subsystems:

  • Plugin Handler: Bridges the C-plugin (Unix socket) to the Log Server (TLS).
  • eBPF Subsystem: Records SSH/TTY login sessions and pkexec scopes via kernel tracepoints (sys_enter_write, sys_enter_execve).
  • Divergence Detector: Detects if sudo ran without the plugin logging it (e.g. plugin was bypassed).
  • Process Sandbox: Enforces kernel-level LSM policy on files, network, and process capabilities.

3. Log Server (Go)

The central ingestor (sudo-logserver). A high-performance, stateless service that verifies Agent identities and persists signed audit chunks to S3 or local storage.

Session Lifecycle

The technical sequence of events for an audited privileged command:

  1. Invocation: User runs sudo dnf update. Sudo loads the plugin.
  2. Plugin Handshake: The plugin connects to the Agent via a protected Unix socket (/run/sudo-logger/plugin.sock).
  3. Context Discovery: The Agent identifies the user via D-Bus and records the parent PID, environment, and TTY path.
  4. Remote Auth: The Log Server is queried. If the command matches a JIT rule, the user is challenged for a "Reason".
  5. Approval: The command is held in a SIGSTOP state until an auditor clicks "Approve" in the Web UI.
  6. Recording: Every byte of TTY input/output is bundled into signed binary chunks and streamed over mTLS.
  7. Risk Scoring: As data arrives, the Server analyzes it against risk-rules.yaml. If a forbidden string (e.g. nc -l) appears, the risk score is instantly elevated.
  8. Finalization: Sudo exits. Final chunks are flushed. The session is closed and indexed for search.

Log Integrity & Ed25519

In a high-security environment, logs must be immutable. sudo-logger uses Ed25519 signatures to provide non-repudiation.

  • Continuous Verification: Every single chunk of data is signed. A missing or reordered chunk will immediately break the cryptographic chain.
  • Hardware Correlation: Signatures are generated using a private key unique to the monitored host, proving the data originated from that specific node.
  • Audit Tooling: The verify-integrity.sh script allows forensic teams to validate log files offline, independent of the Replay Server.

Deployment: RPM Installation

Optimized for RHEL 8/9, AlmaLinux, Rocky Linux, and Fedora.

1. Client (Monitored Host)

dnf install sudo-logger-client
# Configure server = logserver.example.com:9876 in /etc/sudo-logger/agent.conf
systemctl enable --now sudo-logger-agent

2. Server (Control Plane)

dnf install sudo-logger-server sudo-logger-replay
systemctl enable --now sudo-logserver
systemctl enable --now sudo-replay

Data Migration: Local to S3

If you are upgrading from a single-node setup to a distributed cluster, use the built-in migration tool to move recordings to S3.

# Run on the log server
sudo-logserver --migrate \
  --src-path=/var/log/sudoreplay \
  --s3-bucket=audit-logs \
  --s3-endpoint=https://s3.internal

Secret Redaction Deep-Dive

Redaction happens locally on the agent using a Surgical Redactor that masks only the sensitive value while preserving key names (e.g. api_key=****************).

Entropy Detection (KALLE=...)

Any variable assignment where the value is a random string (Hex/Base64) of 24 characters or more is automatically redacted, even if the variable name is unknown.

Gitleaks Patterns

  • AWS Access Keys: AKIA...
  • GitHub PATs: ghp_...
  • Private Keys: PEM-encoded private key blocks (SSH, SSL/TLS).

eBPF Sandbox Policy

Uses eBPF LSM hooks to enforce a deny-list that even root cannot bypass. Scoped via cgroup ID and PID tracking propagated at fork time.

# /etc/sudo-logger/sandbox.yaml
protect:
  files:
    - /etc/sudoers
    - /var/log/sudo-logger
  forbidden:
    - /usr/bin/firewall-cmd
    - /usr/bin/busctl
  processes:
    - sudo-logger-age
    - auditd

JIT (Just-In-Time) Approval System

The JIT system is our proprietary "Four-Eyes" enforcement engine. It ensures that high-risk commands cannot run without active human oversight.

The Workflow (Deep-Dive)

  1. Trap: When a command matches a JIT rule, the Agent immediately sends SIGSTOP to the sudo process group and sets cgroup.freeze=1.
  2. Challenge: The C-plugin detects the freeze and displays a custom TTY banner asking the user for a Justification.
  3. Notify: The Server triggers a webhook (Slack/Mattermost) with a deep-link to the /approvals page.
  4. Release: Upon approval, the Server sends an APPROVAL_GRANTED frame. The Agent resumes the process via SIGCONT and cgroup.freeze=0.
Linger Mode: If a user starts a GUI application (e.g. xterm) and exits the shell, our Agent enters "Linger Mode", keeping the connection alive until the GUI process closes.

Developer: Wire Protocol Spec

All communication uses a proprietary binary protocol over mTLS. Every message has a 5-byte header: [1 byte type][4 bytes length].

// Message Types
0x01 SESSION_START (JSON metadata)
0x02 CHUNK         (Binary data + Stream ID)
0x03 SESSION_END   (Exit code)
0x04 ACK           (Ed25519 Signature)
0x09 HEARTBEAT     (Sent every 400ms)

ACK Signing

The Server signs ACKs using a private Ed25519 key. The signature covers sessionID + sequence + timestamp. This prevents replay attacks where an ACK from one session is used to unfreeze another.

OPA Policy (Rego)

Advanced authorization logic is evaluated via Open Policy Agent. Rules are compiled from the UI into Rego logic. Example of a production rule:

package sudologger.authz

default allow = false

# Allow database-admins to run psql on db-hosts
allow {
    input.user_groups[_] == "db-admins"
    input.host_groups[_] == "database-tier"
    input.command == "/usr/bin/psql"
    not is_weekend
}

is_maintenance {
    time.now_ns() >= time.parse_rfc3339_ns("2026-06-15T22:00:00Z")
}

Visual Sudoers Editor

The Policy → Sudoers tab provides a safe way to manage /etc/sudoers.d/ files across thousands of hosts.

  • Remote Validation: Before saving, the Server sends the config to a representative Agent which runs visudo -c. The save is rejected if syntax is invalid.
  • Snapshot Diff: View side-by-side differences between the current live config and any historical snapshot.
  • Host Pattern Matching: Rules can be targeted using globs (e.g. app-prod-*) or static hostnames.

SIEM Forwarding

The Server can forward audit events (not the full TTY stream) to external SIEMs in real-time. Supported formats:

  • Splunk HEC: Native JSON over HTTPS with index-redirection.
  • CEF (ArcSight): Common Event Format over Syslog.
  • OCSF: Open Cybersecurity Schema Framework for modern lake integration.
