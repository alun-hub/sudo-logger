Session indicators

        

          Badges shown on session cards. Hover a badge for extra detail where available.
        

        

          
BadgeMeaning

          
            

              ebpf
              Session captured by the eBPF TTY recorder instead of the sudo plugin. Appears for
                  privilege escalations via su, screen, tmux
                  or other programs that bypass sudo. The session is fully recorded — the capture
                  path is just different.
            

            

              pkexec
              Privilege escalation via pkexec, captured by the eBPF
                  execve tracepoint. pkexec does not go through the sudo plugin so
                  eBPF is the only capture mechanism for these events.
            

            

              live
              Session is currently running. The recording is still in progress; the replay
                  shows whatever has been captured so far.
            

            

              ▣ screen
              Session contains a terminal recording with full frame history (screen or tmux).
                  The player renders these as a video-like replay rather than a plain text stream.
            

            

              ⚡ event
              pkexec run with no attached TTY — a background privilege escalation with no
                  interactive I/O. The record shows metadata (user, command, exit code, timestamp)
                  but there is no terminal replay.
            

            

              ⏱ network outage
              The agent lost contact with the log server during this session. Session processes
                  were frozen until connectivity was restored so that no activity went unrecorded.
                  The recording is complete but was delayed.
            

            

              ⚠ incomplete
              The session ended without a clean close — the agent was killed or crashed before
                  sending SESSION_END. The recording may be truncated.
            

            

              ⚠ no plugin
              eBPF detected a sudo execution on this host but the audit plugin never logged a
                  matching session. This can mean the plugin is not installed, was bypassed, or
                  the binary was invoked directly. There is no full audit trail — investigate
                  immediately.
            

          
        

      


      

        
How risk scoring works

        

          Every session is evaluated against all rules in
          /etc/sudo-logger/risk-rules.yaml. Each matching rule adds
          its score to the session total (capped at 100). Scores are cached
          alongside each session and only recalculated when the rules file
          changes.
        

        

          

            
Low
0 – 24
          

          

            
Medium
25 – 49
          

          

            
High
50 – 74
          

          

            
Critical
75 – 100
          

        

      


      

        
Rule fields

        

          
FieldDescription

          
            
ScorePoints added to the session when this rule matches (1–100). Multiple matching rules accumulate.

            
IDUnique identifier used internally. Use snake_case — e.g. stop_auditd.

            
ReasonShort label shown in the session info bar, risk badge tooltip, and Anomalies tab.

          
        

      


      

        
Command matching

        
Matched against the full sudo command line — everything after sudo.

        

          
FieldDescription

          
            

              Must contain one of
              The command line must include at least one of the listed substrings.
                  Example: auditd, audit — matches systemctl stop auditd.
            

            

              AND one of
              In addition to the above, the command must also contain at least one of these.
                  Use this to require two independent keywords — e.g. stop and disable
                  to match only destructive audit commands.
            

            

              Command name is one of
              Matches just the program name (basename of argv[0]).
                  More reliable than substring matching for shell interpreters —
                  e.g. bash, sh, zsh, python3.
            

          
        

      


      

        
Shell content matching

        

          Scans what appeared on the terminal (ttyout) for interactive sessions such as
          sudo bash. Useful for detecting commands typed inside a shell —
          these are invisible to command-line matching since sudo only sees bash.
        

        

          
FieldDescription

          
            

              Must contain one of
              At least one of the listed substrings must appear in the terminal output.
                  Matching is case-insensitive and ANSI escape codes are stripped first.
            

            

              AND one of
              The terminal output must also contain at least one of these —
                  use to require two separate indicators before scoring.
            

          
        

      


      

        
Session metadata conditions

        

          
FieldDescription

          
            
Runas userOnly match sessions where sudo ran as this user. Leave blank to match any. Default is root.

            
Min durationOnly match sessions that lasted at least this many seconds. Useful for flagging unusually long interactive shells.

            
Incomplete sessionOnly match sessions that ended without a clean SESSION_END — e.g. the agent was killed.

            
After business hoursOnly match sessions started between 23:00 and 05:59.

          
        

      


      

        
How conditions combine

        

          

            AND
            All conditions in a single rule must be satisfied for the rule to match.
            A rule with both a command pattern and Runas = root only fires
            when both are true.
          

          

            OR
            Items within a tag list are alternatives — any one of them is enough.
            Listing auditd and rsyslog in Must contain one of
            matches either word.
          

          

            OR
            Command matching and shell content matching are alternatives — a rule fires
            if either the command or the content pattern matches (while all
            other conditions still hold).
          

        

      


      

        
Examples

        

          

            
Detect stopping the audit daemon

            
Matches any sudo command that contains both "auditd" and one of "stop" / "disable" / "mask".

            
score: 80
id:    stop_auditd
reason: Audit daemon stopped or disabled
command:
  contains_any: [auditd]
  also_any:     [stop, disable, mask]

          

          

            
Detect interactive root shell

            
Fires when the command basename is a shell interpreter — regardless of arguments.

            
score: 50
id:    root_shell
reason: Direct root shell
command_base_any: [bash, sh, zsh, fish, ksh]

          

          

            
Detect passwd manipulation inside a shell

            
Scans terminal output for commands that modify /etc/passwd or /etc/shadow — visible even though the outer command is just "bash".

            
score: 70
id:    passwd_tamper
reason: Auth file manipulation
content:
  contains_any: [/etc/passwd, /etc/shadow, /etc/sudoers]

          

        

      


      

        
Tips

        

          
Use Command name is one of for shell interpreters — it is more precise than substring matching since it only checks the program name.

          
Shell content matching reads up to 512 KB of terminal output. It is only useful for sessions where a shell was invoked interactively.

          
Scores accumulate across rules. A session matching three rules with scores 30, 40, and 20 gets a total of 90 (Critical). Design scores accordingly.

          
Leave unused fields empty — only filled fields are evaluated. An empty rule (no conditions, no metadata) matches every session.

          
Changes saved in Settings take effect immediately. Cached scores are invalidated automatically on the next page load.

Process Sandbox

        

          The process sandbox uses eBPF LSM hooks to enforce kernel-level restrictions on every sudo session.
          Even a root process with full capabilities cannot bypass these restrictions — they run below capability checks in the kernel.
          Configure the sandbox in Settings → Sandbox. Changes reach all connected agents within 60 seconds.
        

        

          
CategoryBlocksExample

          
            

              Files & directories
              Open for write, truncate, delete, rename, setattr on listed paths. Directories are expanded recursively.
              /etc/sudoers, /var/log/sudo-logger
            

            

              Device nodes
              Open for write / read of raw device nodes (prevents direct disk or memory access).
              /dev/mem, /dev/sda
            

            

              /proc entries
              Write to /proc entries (prevents sysctl changes, sysrq triggers, etc.).
              /proc/sysrq-trigger
            

            

              Unix sockets
              Delete or replace Unix socket files (prevents tampering with agent or D-Bus sockets).
              /run/sudo-logger/plugin.sock
            

            

              Process names
              Send signals to named daemons (prevents kill/stop of audit or security services). Names are matched against /proc/pid/comm — max 15 characters.
              auditd, sshd, systemd-journal
            

          
        

        

          
Propagation: Save in the UI → stored in the shared config backend → agents poll every 60 s → BPF maps updated live. No agent restart required.

          
Local override: If sandbox_config = /path/to/sandbox.yaml is set in agent.conf, that file is loaded at startup. The server-managed config is still fetched and applied on top during the first poll.

          
Violation alerts: Blocked operations are logged as SANDBOX VIOLATION in the agent log and forwarded to the log server as structured SIEM events with action= (e.g. FILE_WRITE, CAPABLE, EXEC_BLOCK), path=, comm=, and session context.

        

      


      

        
Sudoers Management

        
Centrally manage /etc/sudoers.d/sudo-logger-managed across your infrastructure. The UI uses a hierarchical model to simplify configuration deployment.


        

          

            Default
            
Inherited Configuration: The host follows the _default template. Any changes made to the "Main" template are automatically pushed to all hosts with this badge.

          

          

            Modified
            
Host Override: This host has its own specific configuration. It ignores the _default template until the override is removed.

          

          

            ✓
            
In Sync: The agent on the host has successfully applied and verified the latest configuration.

          

          

            ⏳
            
Pending: The host is waiting to fetch the latest configuration (agents poll every 60 seconds).

          

          

            ✖
            
Sync Error: The agent failed to apply the configuration (e.g. visudo -c syntax error). Hover for details.

          

        


        

          
Safety First: Every save is validated with visudo -c on the server and again on the agent before being applied. If validation fails, the change is rejected and the previous configuration is kept.

          
Deployment: Agents poll for new configuration every 60 seconds. You don't need to manually push or restart services.

          
Cleanup: Click "Remove override" to make a host return to following the global template.

        

      


      

        
How to Write Sudoers Rules

        
The visual editor follows standard sudoers syntax but simplifies it into an "Identity-First" card layout.


        

          
            

              Field
              Description & Examples
            

          
          
            

              Identity
              User or Group that the rule applies to. Groups should be prefixed with % (e.g., %wheel, %admins). Users are written as-is (e.g., alun).
            

            

              Hosts
              Where the command can be run. Use ALL for all hosts, or specify a list (e.g., localhost, fedora).
            

            

              Run As
              Which user the command is executed as. Usually (ALL) or (root). You can also specify (ALL : ALL) to allow running as any user/group.
            

            

              Options
              Control execution behavior. NOPASSWD allows running without a password. SETENV allows preserving environment variables.
            

            

              Commands
              The full path to the allowed binary (e.g., /usr/bin/dnf). Use ALL for everything. You can restrict commands by adding arguments (e.g., /usr/bin/systemctl restart nginx).
            

          
        


        

          
Best Practice: Always use full paths for commands (/usr/bin/id instead of just id) to prevent path-injection attacks.

          
Aliases: If you need to reuse lists of users or commands, define them in the Aliases section at the bottom and then use the Alias name in your rules.

          
Global vs Local: Write common rules in the Main template. Use host overrides only for rules that are truly unique to a single machine.

        

      


      

        
JIT Approval — Mattermost/Slack integration

        

          Configure in Settings → JIT Approval. When a user's sudo request is pending,
          a notification is posted to the configured channel with Approve and Deny buttons.
          Clicking a button calls back to this server and instantly resolves the request.
        

        

          
FieldDescription

          
            

              Webhook URL
              Incoming webhook URL for your Mattermost or Slack workspace. Notifications are posted here.
            

            

              Request channel
              Channel where new approval requests are posted (e.g. sudo-logger). Outcome notifications are sent as DMs to the user's handle.
            

            

              Replay Web App URL
              Public base URL of this server, reachable by the Mattermost/Slack server (e.g. http://10.42.0.1). Used to construct the callback URL for interactive buttons.
            

            

              Webhook secret
              
                Required when Webhook URL is set. A secret shared between this server and your webhook integration.
                Used to sign and verify the Approve/Deny button callbacks with HMAC-SHA256, preventing anyone from forging an approval.
                If left empty, interactive buttons are disabled and no callbacks are accepted.
              
            

            

              Mention user
              When enabled, outcome notifications mention the user's handle (@alice) so they are notified in Mattermost/Slack.
            

          
        

        

          
Webhook secret is mandatory for interactive buttons to work. Without it, the callback endpoint rejects all requests and no Approve/Deny buttons appear in notifications.

          
Network access: The Mattermost/Slack server must be able to reach Replay Web App URL/api/approvals/callback. If using an internal IP, enable Allow Untrusted Internal Connections in the Mattermost System Console.

          
Reverse proxy: If this server sits behind an authenticating proxy (e.g. oauth2-proxy), exempt /api/approvals/callback — it is protected by its own HMAC verification and does not need proxy-level auth.

          
UI vs file: In distributed mode all settings are stored in the database. In local mode they are read from /etc/sudo-logger/approval-policy.yaml. Changes saved here take effect within 30 seconds without a server restart.

        

      


      
      

        
OPA JIT Policy

        

          The OPA (Open Policy Agent) policy engine replaces the simple exempt-list with a full rule table.
          Every SESSION_START is evaluated by an embedded Rego module. The result determines
          whether the session is allowed immediately,
          challenged (justification required),
          or hard-denied.
        


        
Decision priority (deny always wins)

        

          

            DENY
            Any matching deny rule vetoes all allow rules. Order within the table does not matter.
          

          

            ALLOW
            If no deny rule matches and at least one allow rule matches → session proceeds without a prompt.
          

          

            CHALLENGE
            No allow rule matches → justification is required and an approval request is created.
          

          

            DEFAULT
            If no rule matches at all, the Default action dropdown applies (allow or challenge).
          

        


        
Rule fields

        

          
FieldFormatMeaning

          
            
Usersglob, @group, comma-sep; * = allMatches the username that ran sudo.

            
Hostsglob, @group, comma-sep; * = allMatches the hostname of the monitored machine.

            
Commandsglob, comma-sep; * = allMatches the full command path (e.g. /bin/bash, /usr/bin/apt*).

            
Runasglob, @group, comma-sep; * = allMatches the target user sudo runs as (usually root).

            
Sys groupsgroup names, comma-sep (AND)User must belong to all listed OS groups. Resolved by the agent via NSS — covers local, SSSD, LDAP, and Active Directory groups transparently.

            
WeekdaysMon–Sun checkboxes; unchecked = any dayRestricts the rule to specific days of the week (server clock, CET).

            
Hour from / Hour to0–23; -1 = no constraintTime window (hour-of-day, exclusive upper bound). Overnight ranges (e.g. 22–06) are supported.

          
        


        
Local policy groups

        

          Under the Groups sub-tab you can define named lists of users, hosts, or runas targets.
          Reference them in rule fields as @groupname.
          This avoids repeating the same member list across multiple rules — update the group once and all rules pick up the change automatically.
        

        

          
Example: SRE on-call window

          

            Define group sre-team → members: alice, bob, sre-*

            Add rule: Users = @sre-team · Weekdays = Mon–Fri · Hour = 20:00–06:00 · Action = allow

            Default action = challenge


            Result: SRE members can sudo without a prompt on weeknight on-call shifts.
            All other users and times require a justification.
          

        


        
Compiled Rego

        

          The Compiled Rego sub-tab shows the exact Rego module that the server passes to OPA.
          Use it to verify that your rules translate as expected, especially for complex group references or overnight time windows.
          The raw Rego field at the bottom lets advanced operators append arbitrary Rego to the generated module.
        


        

          
Sys groups vs local groups: Use Sys groups (rule field) for OS-level group membership (LDAP, AD, local /etc/group). Use Groups (sub-tab) for policy-level aliases like @sre-team that group users by name pattern.

          
Clock: Time and weekday rules use the log server's clock (Europe/Stockholm). The agent's clock is irrelevant.

          
Deny is absolute: A deny rule blocks the session even if an allow rule also matches. There is no way to "override" a deny with a lower-priority allow.

          
Save policy: Changes take effect immediately on the next sudo invocation — no server restart needed.
