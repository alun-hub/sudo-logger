How risk scoring works

Every session is evaluated against all rules in /etc/sudo-logger/risk-rules.yaml. Each matching rule adds its score to the session total (capped at 100). Scores are cached alongside each session and only recalculated when the rules file changes.

Rule fields

FieldDescription
ScorePoints added to the session when this rule matches (1–100). Multiple matching rules accumulate.
IDUnique identifier used internally. Use snake_case — e.g. stop_auditd.
ReasonShort label shown in the session info bar, risk badge tooltip, and Anomalies tab.

Command matching

Matched against the full sudo command line — everything after sudo.

FieldDescription
Must contain one of The command line must include at least one of the listed substrings. Example: auditd, audit — matches systemctl stop auditd.
AND one of In addition to the above, the command must also contain at least one of these. Use this to require two independent keywords — e.g. stop and disable to match only destructive audit commands.
Command name is one of Matches just the program name (basename of argv[0]). More reliable than substring matching for shell interpreters — e.g. bash, sh, zsh, python3.

Shell content matching

Scans what appeared on the terminal (ttyout) for interactive sessions such as sudo bash. Useful for detecting commands typed inside a shell — these are invisible to command-line matching since sudo only sees bash.

FieldDescription
Must contain one of At least one of the listed substrings must appear in the terminal output. Matching is case-insensitive and ANSI escape codes are stripped first.
AND one of The terminal output must also contain at least one of these — use to require two separate indicators before scoring.

Session metadata conditions

FieldDescription
Runas userOnly match sessions where sudo ran as this user. Leave blank to match any. Default is root.
Min durationOnly match sessions that lasted at least this many seconds. Useful for flagging unusually long interactive shells.
Incomplete sessionOnly match sessions that ended without a clean SESSION_END — e.g. the shipper was killed.
After business hoursOnly match sessions started between 23:00 and 05:59.

How conditions combine

AND All conditions in a single rule must be satisfied for the rule to match. A rule with both a command pattern and Runas = root only fires when both are true.
OR Items within a tag list are alternatives — any one of them is enough. Listing auditd and rsyslog in Must contain one of matches either word.
OR Command matching and shell content matching are alternatives — a rule fires if either the command or the content pattern matches (while all other conditions still hold).

Examples

Detect stopping the audit daemon
Matches any sudo command that contains both "auditd" and one of "stop" / "disable" / "mask".
score: 80 id: stop_auditd reason: Audit daemon stopped or disabled command: contains_any: [auditd] also_any: [stop, disable, mask]
Detect interactive root shell
Fires when the command basename is a shell interpreter — regardless of arguments.
score: 50 id: root_shell reason: Direct root shell command_base_any: [bash, sh, zsh, fish, ksh]
Detect passwd manipulation inside a shell
Scans terminal output for commands that modify /etc/passwd or /etc/shadow — visible even though the outer command is just "bash".
score: 70 id: passwd_tamper reason: Auth file manipulation content: contains_any: [/etc/passwd, /etc/shadow, /etc/sudoers]

Tips

Use Command name is one of for shell interpreters — it is more precise than substring matching since it only checks the program name.
Shell content matching reads up to 512 KB of terminal output. It is only useful for sessions where a shell was invoked interactively.
Scores accumulate across rules. A session matching three rules with scores 30, 40, and 20 gets a total of 90 (Critical). Design scores accordingly.
Leave unused fields empty — only filled fields are evaluated. An empty rule (no conditions, no metadata) matches every session.
Changes saved in Settings take effect immediately. Cached scores are invalidated automatically on the next page load.
