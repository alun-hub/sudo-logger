Redesigning the Sudoers UI

We compared the current Sudoers editor with three modern, simplified design approaches to improve user experience.

Current UI (Pain Points)
🔴 UX Issues identified:
  • Uses browser prompt() dialogs to add users/groups/commands (very legacy/jarring).
  • Advanced settings (NOEXEC, SETENV, RunAs) are hidden inside expandable <details> on every single card.
  • Long scrolling list of large cards; hard to scan when there are many rules.
  • Syntax validation errors are only shown after trying to save.

Proposed Design Directions

Select the direction you prefer for the visual layout:

A

Inline Modernization (Refined Cards)

Keep the current card list layout but replace all browser prompts with slick inline inputs, tag editors for commands, and expose common advanced settings directly without a details collapse.

B

Master-Detail Split Panel (Highly Recommended)

A double-column layout inside the main editor: a clean, compact list/table of active rules on the left, and a dedicated, rich editing sidebar/inspector panel on the right for the selected rule. Ideal for scaling.

C

Structured Rule Grid (Unified Table)

Flatten rules into a modern interactive spreadsheet/table. Principal, Commands, RunAs, and Flags each get columns, allowing admins to edit values inline like a grid without large bulky cards.

Core UX Enhancements (Applicable to all options)

⚡ Dynamic Tag Input for Commands
Instead of prompt dialogs, typing in a text field and pressing Enter or , instantly creates a command chip. Autocomplete suggests common system paths (e.g. /usr/bin/apt, /bin/systemctl).
🔍 Rule-Level Search Filter
A quick filter bar above the rules list lets admins type a username or command to instantly isolate matching rules, preventing endless scrolling on complex hosts.
✅ Live Syntax & Validation Previews
As rules are modified in the visual editor, a live preview panel translates them into raw sudoers syntax in real-time, showing validation alerts immediately before saving.
💡 Quick Presets (Templates)
Buttons for common permission setups like "Read-only Operator", "Service Administrator", or "Full Root Bypass" to automatically populate rules with recommended defaults.
